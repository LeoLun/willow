# add-automation-feature

## Summary

为桌面端增加“自动化”能力，允许用户为某个工作空间配置一段自定义提示词，并通过前端快捷计划方式或自定义 cron 表达式定义定时触发器。应用需要提供自动化列表页、创建入口、空状态引导，以及自动化相关的数据库、主进程调度与 AI tools CRUD 能力。

## Problem

当前应用已经有“自动化”页面入口，但没有自动化数据模型、调度执行链路、列表 UI、AI 可调用工具或掉线补偿机制。用户无法把固定提示词绑定到工作空间并周期性执行，前端也缺少对常见周期的低门槛配置方式，关闭进程期间错过的任务也无法补跑。

## Goals

- 提供自动化记录与触发器的持久化存储。
- 允许每条自动化绑定一个工作空间和一段用户自定义提示词。
- 首期只支持定时触发，但数据结构和运行时架构要为未来邮件等触发器留出扩展空间。
- 前端创建流程支持“每天 xx 点”“每小时”“每周 周几 xx 点”“自定义 cron”四种计划输入方式。
- 提供基于 shadcn-vue 的自动化页面、空状态和创建弹窗。
- 自动化相关 UI 必须遵守仓库根 `DESIGN.md`，保持桌面工作台、非营销化的视觉与组件表达。
- 使用 `node-cron` 与标准 cron 表达式调度自动化，不引入自定义内存定时器。
- 当应用重启后发现错过执行时，只补执行最后一次应运行的计划点。
- 将自动化的增删查改封装为 AI 可调用 tools。

## Non-Goals

- 本期不实现邮件、Webhook 或文件系统事件等其他触发器。
- 本期不支持自动化链式编排、条件分支、重试策略配置或并发优先级配置。
- 本期不要求提供复杂日历构建器或图形化日历面板，但需要提供常见周期快捷选项与自定义 cron 输入。
- 本期不定义自动化执行结果分析报表，仅要求记录满足补偿执行和列表展示的必要信息。

## Success Criteria

- 用户可以在自动化页查看已有自动化列表，并在空状态下被引导创建第一条自动化。
- 用户可以通过“添加自动化”入口，用快捷计划方式或自定义 cron 创建一条绑定工作空间、触发器与提示词的自动化。
- 自动化页面、空状态和创建弹窗在视觉层级、布局骨架和组件使用上符合仓库根 `DESIGN.md`。
- 主进程在启动时恢复所有启用中的自动化调度，并能在关闭期间漏跑时只补最后一次。
- AI 可以通过工具查询、创建、更新、删除自动化。
- 后续新增触发器类型时，不需要推翻本次自动化实体边界。

## Viable Approaches

### Approach A: 单表承载自动化与触发器配置

将自动化基础字段、触发器类型、cron 配置和未来扩展字段都放在一张表里。

优点：
- 建模简单，首期实现快。

缺点：
- 邮件等未来触发器会让单表快速膨胀。
- 触发器配置与自动化主体耦合过高，不利于后续扩展和校验。

### Approach B: 自动化主体、触发器、执行记录分离

使用自动化主表承载用户可见配置，触发器表承载 `type` 与具体配置，执行记录表承载计划执行与补偿执行信息。

优点：
- 对未来触发器扩展最友好。
- 补偿执行、状态展示和 AI tools 输入输出更清晰。
- 能把“自动化定义”和“执行历史”职责分离。

缺点：
- 首期表结构和服务层稍复杂。

### Approach C: JSON 配置中心化存储

将自动化定义与触发器配置主要以 JSON 文本持久化，只保留少量索引字段。

优点：
- 扩展字段灵活，迁移少。

缺点：
- 查询、校验、迁移和 UI 展示不稳定。
- 不适合作为后续自动化能力的长期基础。

## Recommendation

采用 Approach B。它最符合“首期只支持定时触发，但必须为未来触发器留扩展空间”的约束，也能自然支撑错过执行补偿、执行列表状态、以及 AI tools 的结构化读写。
