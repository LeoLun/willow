# context-compression

## Summary

当当前会话上下文规模接近模型上下文窗口上限、继续发送会导致上下文不足时，系统自动对较早历史做压缩，将其转化为结构化摘要与索引，保留最近若干轮完整消息，确保 Agent 在长对话中持续可用。

## Problem

当前 Agent 启动时会将会话全部历史消息加载并逐条传递给模型（`agent.replaceMessages(history)`），不做任何截断或压缩：

- 随着对话轮数增长，历史消息 token 总量必然超过模型 `contextWindow` 上限。
- 超限后模型会返回 400 错误或截断最旧的消息，导致 Agent 丢失早期上下文。
- 用户只能手动创建新会话来规避，打断连续工作流。
- 没有 token 估算或逼近预警，系统无法在当前会话上下文不足前主动处理。
- `packages/core/src/context/README.md` 已预留了"上下文压缩"和"摘要与索引"的设计占位，但尚无实现。

## Goals

- 每次构建发送消息前，根据模型 `contextWindow` 估算系统提示词、历史摘要、完整历史和本轮用户输入的 token 占用。
- 当估算 token 总量超过安全阈值（默认 `contextWindow * 0.8`）或保守判断本轮发送会导致上下文不足时触发压缩。
- 压缩策略：保留最近 3-5 轮完整消息，将更早的消息压缩为一个结构化摘要块，附带关键信息索引。
- 压缩摘要作为 `CoreAgent` 的系统上下文注入，而不是写入普通聊天消息列表。
- 压缩状态持久化到主进程可复用的数据结构中；用户可见的原始聊天历史不因压缩被删除。
- 压缩过程在 `AgentService` 构建默认 Agent 前完成，对会话发送流程透明。
- 支持在 UI 层提示用户"上下文已压缩"，让用户了解历史信息被摘要化处理。
- 遵守项目现有的 `@mariozechner/pi-agent-core` Agent 协议，不修改 Agent 核心。

## Non-Goals

- 本次不改 `@mariozechner/pi-agent-core` 内部消息管理逻辑。
- 本次不实现向量化或 RAG 式的长期记忆检索。
- 本次不修改 chat UI 的整体布局、输入器或消息渲染。
- 本次不引入外部摘要模型或远程摘要服务；使用当前会话选定模型配置创建隔离的辅助摘要 Agent。
- 本次不处理跨会话的上下文继承或会话合并。
- 本次不要求提供手动压缩按钮或上下文用量仪表盘。

## Success Criteria

- 当估算的 token 使用量超过 `contextWindow * 0.8`，或当前会话上下文不足以容纳本轮发送的安全余量时，系统自动触发压缩。
- 压缩后，发送给模型的系统上下文、摘要、保留历史和本轮用户输入总 token 量降至 `contextWindow * 0.6` 附近；若极长最近窗口导致无法达到 60%，系统仍必须保证不超过安全阈值。
- 压缩产生的摘要块包含：每轮用户问题要点、Agent 完成的关键操作、关键决策或发现。
- 保留最近 3-5 轮完整消息不动，确保当前工作上下文不被压缩。
- 压缩不应无限阻塞正常消息发送流程；摘要生成有超时和失败降级。
- 未触发压缩时，消息构建行为与现有逻辑完全一致。
- UI 在触发压缩后显示"上下文已自动压缩"提示（非阻塞 toast 或消息区内提示条），且不改变聊天页整体布局。
- 最终实现通过 `pnpm lint` 和 `pnpm build` 检查。

## Viable Approaches

### Approach A: 纯规则截断 + 简单摘要头

在 token 超限时，直接丢弃最早的消息，仅保留一段固定文本"较早对话已省略"。

优点：
- 实现最快，无额外 LLM 调用。
- 不会引入摘要失败路径。

缺点：
- 彻底丢失早期上下文，Agent 无法回忆早期决策。
- 用户体验差，和手动清空会话差别不大。

### Approach B: LLM 摘要 + 渐进式索引（推荐）

在 token 逼近阈值时，将较早消息发送给隔离的辅助摘要 Agent 做一次摘要调用，输出结构化摘要（每轮要点 + 关键操作 + 关键决策），作为系统上下文注入。最近 N 轮保留不动。

优点：
- 保留早期上下文的核心信息，Agent 可以回溯关键决策。
- 摘要格式固定，模型可以稳定理解。
- 符合 `packages/core/src/context/README.md` 已规划的"摘要与索引 + 渐进式披露"方向。

缺点：
- 需要一次额外的 LLM 调用，增加延迟。
- 摘要质量依赖模型能力，极端情况下摘要可能遗漏关键信息。

### Approach C: 滑动窗口 + 用户手动触发

不自动压缩，在 UI 上显示上下文用量指示器，让用户手动触发"压缩上下文"。

优点：
- 用户完全可控，不会在关键时刻被打断。
- 实现简单，不需要自动化判断逻辑。

缺点：
- 增加认知负荷，用户需要时刻关注用量。
- 用户不触发会导致 Agent 报错，体验割裂。
- 和项目"降低用户认知负荷"的设计目标不符。

## Recommendation

采用 **Approach B（LLM 摘要 + 渐进式索引）**。

理由：
- 这是 `context/README.md` 已规划的方向，与项目设计一致。
- 自动化处理符合 Willow"降低认知负荷"的使命。
- 保留早期上下文信息的价值远超额外 API 调用的成本。
