# context-compression Specification

## ADDED Requirements

### Requirement: 自动检测当前会话上下文不足

系统 MUST 在每次发送用户消息前估算当前会话将提交给模型的上下文占用，并在上下文不足前触发压缩。

#### Scenario: 未达到阈值时保持现有行为

- **前提** 当前会话的系统提示词、历史消息、历史摘要和本轮用户输入的估算 token 总量低于安全阈值
- **当** 用户发送消息
- **那么** 系统按现有方式把完整历史加载到主 Agent
- **并且** 不创建或更新上下文摘要
- **并且** 不向 renderer 发送上下文压缩提示

#### Scenario: 达到安全阈值时触发压缩

- **前提** 当前会话的估算 token 总量超过当前模型可用上下文的 80%
- **当** 用户发送消息
- **那么** 系统在主 Agent 执行本轮 prompt 前触发上下文压缩
- **并且** 压缩完成后再构建本轮主 Agent 历史

#### Scenario: 本轮输入会导致上下文不足时触发压缩

- **前提** 历史消息本身尚未超过安全阈值
- **并且** 加上本轮用户输入后会超过安全阈值或可用上下文
- **当** 用户发送消息
- **那么** 系统仍然触发上下文压缩
- **并且** 压缩判断必须包含本轮用户输入的估算占用

#### Scenario: 输出 token 需要预留上下文空间

- **前提** 当前模型配置包含 `contextWindow` 和 `maxTokens`
- **当** 系统计算可用上下文
- **那么** 系统必须为模型输出预留空间
- **并且** 压缩触发判断不得把完整 `contextWindow` 都用于输入历史

### Requirement: 压缩早期历史并保留最近完整轮次

系统 MUST 将较早会话历史压缩为结构化摘要，同时保留最近若干轮完整消息供主 Agent 直接使用。

#### Scenario: 保留最近完整轮次

- **前提** 当前会话包含多轮 user / assistant 交互
- **当** 系统触发上下文压缩
- **那么** 系统保留最近 3-5 轮完整交互
- **并且** 保留范围内的消息顺序和内容不被改写

#### Scenario: 按用户消息边界切分压缩范围

- **前提** 当前会话包含多条用户消息和对应 Agent 回复
- **当** 系统选择需要压缩的历史
- **那么** 压缩边界优先落在用户消息轮次边界
- **并且** 系统不得在同一轮用户请求与其主要 Agent 回复之间任意截断

#### Scenario: 摘要包含关键上下文

- **前提** 被压缩历史包含用户目标、Agent 操作、文件路径、命令结果、错误和决策
- **当** 系统生成摘要
- **那么** 摘要必须包含用户问题要点、Agent 完成的关键操作、关键决策或发现、待跟进事项
- **并且** 摘要必须保留仍可能影响后续工作的具体路径、模块名、配置值和错误信息

#### Scenario: 压缩后重新校验预算

- **前提** 系统已经生成或复用历史摘要
- **当** 系统准备提交主 Agent 历史
- **那么** 系统必须重新估算摘要、最近完整窗口和本轮输入的 token 占用
- **并且** 结果应降至目标预算附近
- **并且** 若无法降至目标预算，系统必须继续缩小最近窗口或进入明确降级路径

### Requirement: 持久化压缩状态但保留原始聊天历史

系统 MUST 持久化可复用的上下文摘要状态，并且 MUST NOT 用摘要替换用户可见的原始聊天历史。

#### Scenario: 原始消息仍可查看

- **前提** 某会话已经触发上下文压缩
- **当** renderer 请求该会话历史
- **那么** 系统仍返回原始聊天消息
- **并且** 用户可见聊天记录不包含用于模型的摘要系统上下文

#### Scenario: 下次发送复用已有摘要

- **前提** 某会话已经存在上下文摘要状态
- **并且** 用户继续发送新消息
- **当** 系统再次判断需要压缩
- **那么** 系统复用已有摘要作为压缩基础
- **并且** 只对摘要游标之后且不在最近完整窗口内的新增历史做增量压缩

#### Scenario: 删除会话清理摘要状态

- **前提** 某会话存在上下文摘要状态
- **当** 用户删除该会话
- **那么** 系统删除该会话的原始消息
- **并且** 系统也删除对应上下文摘要状态

### Requirement: 通过系统上下文注入摘要

系统 MUST 将压缩摘要作为主 Agent 的系统上下文使用，而不是作为普通聊天消息写入会话。

#### Scenario: 摘要不进入普通消息列表

- **前提** 系统已为会话生成压缩摘要
- **当** 主 Agent 构建本轮上下文
- **那么** 摘要被注入系统上下文或等价只读上下文块
- **并且** 摘要不会作为 user 或 assistant 消息写入 `session_messages`

#### Scenario: 主 Agent 只接收最近完整历史

- **前提** 系统已触发上下文压缩
- **当** 系统调用主 Agent 的历史替换逻辑
- **那么** 传入的是最近完整窗口内的消息
- **并且** 较早历史通过摘要上下文提供给模型

#### Scenario: 未压缩路径不改变系统提示

- **前提** 当前会话未触发上下文压缩且不存在需要注入的摘要
- **当** 系统构建 `CoreAgent`
- **那么** 系统提示词和历史消息注入行为与现有流程一致

### Requirement: 摘要生成隔离于主对话流程

系统 MUST 使用隔离的辅助摘要 Agent 或等价机制生成摘要，避免污染主会话状态。

#### Scenario: 摘要请求不写入聊天历史

- **前提** 系统正在为会话生成压缩摘要
- **当** 辅助摘要 Agent 完成请求
- **那么** 摘要请求和摘要回复不作为普通会话消息持久化
- **并且** 不出现在聊天 renderer 的消息列表中

#### Scenario: 摘要 Agent 不启用工具

- **前提** 当前聊天 Agent 可使用文件、命令、搜索或自动化工具
- **当** 系统创建辅助摘要 Agent
- **那么** 辅助摘要 Agent 不启用这些工具
- **并且** 摘要生成不会触发工具审批流程

#### Scenario: 摘要失败不污染主 Agent 状态

- **前提** 辅助摘要 Agent 调用失败、超时或返回空摘要
- **当** 系统继续处理本轮发送
- **那么** 主 Agent 的消息状态不包含失败的摘要请求
- **并且** 系统按失败降级规则决定继续发送或缩小上下文

### Requirement: 压缩失败时可降级继续

系统 MUST 对摘要失败、超时和最近窗口过大的情况提供明确降级路径。

#### Scenario: 压缩失败但完整历史仍可发送

- **前提** 系统触发压缩
- **并且** 摘要生成失败
- **并且** 完整历史加本轮输入仍未超过可用上下文
- **当** 用户发送消息
- **那么** 系统允许本轮发送继续使用完整历史
- **并且** 系统记录压缩失败原因

#### Scenario: 压缩失败且完整历史不可发送

- **前提** 系统触发压缩
- **并且** 摘要生成失败
- **并且** 完整历史加本轮输入会超过可用上下文
- **当** 用户发送消息
- **那么** 系统必须保留最近完整窗口继续发送
- **并且** 注入明确的降级上下文说明，告知较早历史因压缩失败可能不完整
- **并且** renderer 必须收到可理解的降级提示

#### Scenario: 最近窗口本身过大

- **前提** 最近 3-5 轮完整消息本身已经超过安全阈值
- **当** 系统尝试压缩
- **那么** 系统优先保留最新消息
- **并且** 可以缩小保留轮次
- **并且** 必须避免向模型提交超过可用上下文的输入

### Requirement: 通知用户上下文已自动压缩

系统 MUST 在当前会话触发上下文压缩或降级时向用户提供非阻塞反馈。

#### Scenario: 当前会话显示压缩提示

- **前提** 用户正在查看某个会话
- **并且** 该会话本轮发送触发上下文压缩并成功
- **当** renderer 收到压缩通知
- **那么** renderer 显示“上下文已自动压缩”或等价中文提示
- **并且** 提示不阻塞用户继续输入或查看消息

#### Scenario: 非当前会话不显示提示

- **前提** renderer 收到某个会话的上下文压缩通知
- **并且** 用户当前查看的是另一个会话
- **当** renderer 处理该通知
- **那么** 当前页面不显示压缩 toast
- **并且** 当前会话消息状态不被改变

#### Scenario: 降级时提示历史可能不完整

- **前提** 上下文压缩失败并进入降级发送
- **当** renderer 收到降级通知
- **那么** renderer 使用非阻塞提示告知较早历史可能不完整
- **并且** 不改变聊天页整体布局
