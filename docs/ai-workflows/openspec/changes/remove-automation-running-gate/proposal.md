# remove-automation-running-gate

## Summary

移除自动化执行链路中会跨进程残留的“正在执行”判断和数据库持久化运行中状态，不再让 `automation_runs.status = running` 或同类数据库状态阻止下一次执行。自动化可以继续保留运行记录与最终结果，但执行开始后即使应用进程被停止，下一次手动或定时触发也必须能够重新启动。

## Background

当前立即执行自动化时，用户可能看到错误：

`Error invoking remote method 'RUN_AUTOMATION_NOW': Error: 自动化正在执行中`

现有实现同时使用内存集合 `runningAutomationIds` 和数据库 `automation_runs.status = running` 判断同一自动化是否正在执行。内存状态在进程退出后会消失，但数据库中的 `running` 记录会保留。如果执行过程中应用停止、崩溃或会话被中断，后续再次执行同一自动化时会因为历史 running 记录被永久拦截，用户无法恢复执行。

## Goals

- 不再通过数据库中的 `running` 自动化运行记录阻止下一次执行。
- 不再把“自动化正在执行”作为需要跨进程恢复的业务状态保存。
- 手动立即执行在历史运行记录残留为 `running` 时仍然可以再次启动。
- 定时执行或启动补偿在历史运行记录残留为 `running` 时仍然可以继续触发。
- 保留自动化运行记录的完成/失败摘要能力，便于页面展示最近运行结果。
- 后台执行失败或会话发送失败时仍记录 `failed` 与错误信息。
- 避免同一按钮的单次快速重复点击造成明显的重复请求；这种 UI 层加载态可以保留为短生命周期交互保护。

## Non-Goals

- 不新增自动化运行队列、取消执行、重试策略或并发上限配置。
- 不新增自动化运行历史详情页。
- 不改变自动化创建、编辑、删除、启停和 cron 配置语义。
- 不改变聊天会话的运行中/停止语义。
- 不要求迁移或清理用户已有数据库中的历史 `running` 记录；实现可以选择忽略它们或在兼容处理时标记为失败。

## Approaches

### Approach A: 启动时清理历史 running 记录

应用启动时把所有 `automation_runs.status = running` 的历史记录批量更新为 `failed` 或 `interrupted`，继续保留数据库 running 查询作为并发门禁。

优点：

- 改动面较小。
- 历史运行记录语义更完整。

缺点：

- 仍然保留数据库状态作为执行门禁，问题模型没有消失。
- 如果执行中停止发生在本次启动之后、下次启动清理之前，用户仍会遇到短期卡死。
- 需要定义新的中断状态或把未知中断伪装成失败。

### Approach B: 移除数据库 running 门禁，仅保留进程内短生命周期保护

执行前不再查询 `automation_runs.status = running` 来拒绝启动。数据库运行记录只作为展示和审计数据；同一进程内可以保留 `runningAutomationIds` 之类短生命周期保护，用于避免一次执行还没完成时同一进程重复启动同一自动化。

优点：

- 直接解决进程停止后历史 running 记录永久阻塞的问题。
- 保留当前进程内的重复触发保护，降低误点产生重复会话的概率。
- 改动范围可控，不需要立即重构运行记录表。

缺点：

- 仍然存在进程内“正在执行”概念，但它不会跨进程保留。
- 如果进程内任务卡住且未释放内存保护，本次进程生命周期内仍可能阻止重复触发。

### Approach C: 完全移除自动化并发门禁

自动化执行前不检查内存集合，也不检查数据库 running 记录。每次手动点击或 cron 触发都创建独立会话并执行。

优点：

- 最彻底地消除“正在执行中”阻塞。
- 行为简单，任何触发都可尝试执行。

缺点：

- 快速重复点击或 cron 重入会产生多个并发会话。
- 可能导致重复消耗模型调用和更多 UI 噪音。

## Recommendation

采用 Approach B，并在任务中明确边界：数据库中的运行状态不得作为执行门禁；自动化服务可以保留进程内短生命周期保护，但必须保证应用重启、停止后残留的运行记录不会影响下一次执行。如果实现阶段发现进程内保护也会因会话停止无法释放，应进一步收敛为仅由 renderer 按钮 loading 防抖，回到 Approach C。

## Success Criteria

- 当某条自动化存在历史 `automation_runs.status = running` 记录时，点击“立即执行”仍能启动新的执行。
- 应用在自动化执行过程中被停止后，重新打开应用并再次执行同一自动化不会再提示“自动化正在执行中”。
- 定时触发和漏跑补偿不会因为历史 running 记录而跳过本次执行。
- 新执行仍会创建会话、发送自动化 prompt，并在成功或失败后更新运行记录。
- UI 可以继续在一次点击请求期间禁用按钮或展示 loading，但不会因为数据库残留状态永久禁用或报错。
- 现有自动化列表、详情页最近运行摘要仍可展示最近运行结果。
