# add-workspace-init-command

## 摘要

为 Willow 增加一套“工作空间 Agent”能力：在项目工作空间中通过内置 `/init` 初始化或改进根目录 `AGENTS.md`，并让“对话”作用域把各工作空间视为可发现、可显式指定、可按需自动调用的子 Agent。`AGENTS.md` 顶部 YAML frontmatter 中的 `name` 与 `description` 将像 skill 的元信息一样被加载，用于工作空间 Agent 的发现和调度。

## 问题

当前仓库已经约定使用 `AGENTS.md` 作为工作空间上下文的重要输入，工作空间设置页也会直接读写该文件。但系统仍缺少一套完整机制，把“有 AGENTS.md 的工作空间”升级为可被对话调用的工作空间 Agent，带来几个问题：

- 新建或接入工作空间后，用户不知道如何快速得到一份可用的 `AGENTS.md`
- 已有 `AGENTS.md` 往往只覆盖仓库协作规则，没有明确可用于调度的 frontmatter `name`、`description`、触发语义和工作空间索引
- 现有技能发现机制只加载 `.agents/skills`，不适合承载“内置、始终可用、与工作空间生命周期绑定”的初始化命令
- 对话作用域与项目工作空间作用域已经存在清晰边界，但当前对话还不能把工作空间作为“可调用子 Agent”来协作
- 对话输入区虽然已有 slash 资源机制，但还不能显式指定某个工作空间 Agent 来承接任务

## 目标

1. 新增内置 `/init` 能力，在项目工作空间中可发现、可调用
2. `/init` 只属于工作空间，不在“对话”作用域中显示或执行
3. `/init` 执行时自动分析当前工作空间，并通过 LLM 创建或改进工作空间根目录 `AGENTS.md`
4. `AGENTS.md` 必须在文件顶部提供 YAML frontmatter，并包含可供调度的 `name` 与 `description` 元信息
5. 对话作用域加载各工作空间 `AGENTS.md` 的 frontmatter `name` 与 `description`，像加载 skill 摘要一样建立“可调用工作空间 Agent”列表
6. 对话可以根据用户请求和工作空间描述，按需自动调用某个工作空间子 Agent 执行任务
7. 对话页必须作为主 Agent，工作空间必须作为真正独立的子 Agent 会话来执行，而不是仅切换主 Agent 的 cwd
8. 对话输入区可以通过 `/` 显式指定某个工作空间子 Agent
9. 对话调用工作空间子 Agent 必须建模为一个 tool，有独立的 tool 调用记录和 UI 渲染效果
10. 用户点击该 tool UI 后，系统应跳转到对应的子 Agent 会话
11. `/init` 的触发、发送中状态与结果回流在 UI 上应与普通 skill 调用保持一致的交互感知
12. 子 Agent 回调给主 Agent 的内容必须仅包含最终执行结果、失败原因或停止结论，不回传其中间工具调用、审批细节和过程消息
13. 生成内容保留用户提供的整体提示结构，并优化表达与可执行性
14. 输出文件应包含：
   - 工作空间名称
   - 工作空间作用、已有技能、必须文件索引
   - 必填 frontmatter 字段：`name`、`description`
   - 正文可继续使用中文“名称”“描述”等章节辅助人类阅读，但调度解析以 frontmatter 为准
   - 明确“何时触发，以及触发效果”，作为对话功能调用此工作空间的主要触发机制
15. 生成内容控制在约 150 行

## 非目标

- 本次不把 `/init` 做成普通 `.agents/skills` 目录中的可编辑技能包
- 本次不处理全局 `AGENTS.md`、用户主目录说明或对话作用域的专属初始化
- 本次不扩展为通用“任意模板文件生成器”
- 本次不改变现有工作空间设置页手动编辑 `AGENTS.md` 的能力
- 本次不要求把所有工作空间任务都强制改为多 Agent 执行
- 本次不引入任意层级的工作空间嵌套调度图

## 成功标准

- 工作空间会话中打开资源选择时可以发现 `/init`
- 对话作用域中不能发现或调用 `/init`
- 当工作空间根目录不存在 `AGENTS.md` 时，`/init` 会创建它
- 当工作空间根目录已存在 `AGENTS.md` 时，`/init` 会在保留有效仓库约束的前提下改进它，而不是盲目覆盖
- `/init` 的生成过程必须真实调用 LLM，而不是仅靠本地模板或字符串拼接完成
- 生成后的 `AGENTS.md` 顶部包含 `name` 与 `description` frontmatter，并明确覆盖工作空间名称、职责、技能、关键文件索引与触发说明，且整体长度约 150 行
- 对话作用域能够读取工作空间 Agent 的 frontmatter `name` 与 `description`
- 对话既可以自动选择合适的工作空间子 Agent，也可以通过 `/` 显式指定某个工作空间子 Agent
- 被调用的工作空间 Agent 必须以独立子 Agent 会话执行，并仅将最终结果回流给对话主 Agent
- 对话中的工作空间委派会显示为一个独立 tool 调用 UI
- 用户点击该 tool 调用 UI 后可以跳转到对应的子 Agent 会话
- 用户触发 `/init` 时，在 UI 上看到的调用方式、执行中状态和结果展示与普通 skill 调用保持一致
- 主会话默认不展示子 Agent 的中间工具调用与过程细节，这些内容只保留在子 Agent 会话中

## 可行方案

### 方案 A：把 `/init` 做成普通工作空间技能

优点：

- 复用现有 `.agents/skills` 加载与 slash 选择链路
- 数据模型简单

缺点：

- 用户仍需要先有技能目录或模板，才能生成 `AGENTS.md`
- “初始化工作空间”本身依赖工作空间内容分析，不适合作为普通可搬运技能
- 难以保证它在所有工作空间中都始终存在

### 方案 B：把 `/init` 做成内置工作空间命令，并把工作空间建模为子 Agent（推荐）

优点：

- 与“工作空间初始化”语义一致，开箱即用
- 可以直接复用工作空间路径、文件树、已有 `AGENTS.md` 和技能扫描结果
- 更容易精确控制“工作空间可见、对话不可见”的边界
- 可以把 `AGENTS.md` 进一步作为工作空间 Agent 的身份卡和路由摘要

缺点：

- 需要在资源选择和执行链路里新增一类内置命令或等价能力
- 需要额外定义与普通技能不同的展示与执行契约
- 需要设计主 Agent 与子 Agent 的委派边界

### 方案 C：只在创建工作空间时自动写入默认 `AGENTS.md`

优点：

- 实现范围最小
- 与现有 `WorkspaceService` 写入 `AGENTS.md` 的能力接近

缺点：

- 无法支持用户在后续重新分析并改进已有工作空间
- 不满足“作为可调用功能提供给对话”的需求
- 无法根据后续仓库演进重建更完整的索引

### 方案 D：对话直接把工作空间当作普通技能项

优点：

- 可以复用现有技能列表和 slash UI 交互习惯
- 用户理解成本较低

缺点：

- 工作空间并不是“提示片段”，而是带独立根目录和 `AGENTS.md` 的执行单元
- 容易混淆“给主 Agent 的技能提示”和“需要实际派发任务的子 Agent”
- 无法清楚表达自动调度、显式指定和执行结果回流等多 Agent 语义

## 推荐

采用 **方案 B：内置工作空间命令 + 工作空间子 Agent 调度**。

理由：

- 用户需求本质上是“让系统在当前工作空间里执行一次初始化动作”，并让对话能消费这份初始化结果
- 现有系统已经区分工作空间与对话作用域，适合在这里加一条清晰边界
- `AGENTS.md` 可以同时承担工作空间本地协作说明与对话路由摘要两种职责，形成初始化与调用的一体化模型
