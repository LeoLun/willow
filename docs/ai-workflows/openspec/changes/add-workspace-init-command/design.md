# add-workspace-init-command 设计文档

## 背景

当前 Willow 已具备以下基础能力：

- 工作空间设置页通过 `WorkspaceService` 直接读写工作空间根目录 `AGENTS.md`
- `loadSkills()` 只负责发现全局与工作空间 `.agents/skills` 技能
- `SkillService` 在没有工作空间时，会过滤掉工作空间作用域技能
- 系统提示词、项目上下文和 `AGENTS.md` 已经是 Agent 构建的一部分
- sender 已有 slash 资源面板和结构化资源发送链路

因此，这次需求并不是“再新增一个普通技能”，而是补齐一套工作空间 Agent 机制：

- 在工作空间内通过 `/init` 自动生成或改进 `AGENTS.md`
- 在对话作用域中把带有有效 `AGENTS.md` frontmatter 的工作空间发现为“可调用子 Agent”
- 支持自动调度与显式 slash 指定

## 方案选择

采用提案中的推荐方案：把 `/init` 设计为**内置工作空间命令**，并把工作空间抽象为**可被对话调用的子 Agent**。

关键原因：

- `/init` 不依赖 `.agents/skills` 先存在，适合首次初始化场景
- 它的可见性与执行前提天然绑定工作空间，不应出现在对话作用域
- `AGENTS.md` 是持久化工作空间文件，属于工作空间治理能力，而非单次提示增强
- `AGENTS.md` 还能进一步成为工作空间 Agent 的身份卡与路由描述

## 用户体验

### 工作空间中的发现方式

- 用户在项目工作空间会话中打开 slash 资源选择器时，可以看到 `/init`
- `/init` 在资源面板中应作为内置命令或等价分组项出现，并带有简短说明，例如“分析当前工作空间并创建或改进 AGENTS.md”
- 当当前会话属于 `conversation` 作用域时，资源面板不得显示 `/init`

### 对话中的发现方式

- 用户在对话作用域中打开 slash 资源选择器时，可以看到可用的工作空间 Agent 列表
- 每个工作空间 Agent 项至少展示从 `AGENTS.md` frontmatter 解析出的 `name` 与 `description`
- 若某个工作空间 `AGENTS.md` 不完整，则该工作空间不应被展示为可自动调度目标，系统可提示用户先执行 `/init`

### `/init` 执行效果

选择 `/init` 后，本轮请求应驱动系统执行以下动作：

1. 识别当前工作空间根路径
2. 读取已有 `AGENTS.md`（如果存在）
3. 扫描工作空间关键信息：
   - 工作空间名
   - 主要目录与用途
   - 可见工作空间技能列表
   - 关键约束文件，如 `AGENTS.md`、`DESIGN.md`、OpenSpec 路径、包管理与主要应用入口
4. 组装一段稳定的初始化提示词，引导 Agent 通过 LLM 生成新的 `AGENTS.md`
5. 将 LLM 结果写回工作空间根目录 `AGENTS.md`
6. 向用户返回简洁确认，说明是“已创建”还是“已改进”

这里必须强调：

- `/init` 不允许退化为“本地模板直接拼接后写文件”
- 工作空间分析结果只能作为提示词输入或结构化上下文，最终文本必须由 LLM 生成或重写

### `/init` 的 UI 一致性

虽然 `/init` 是内置命令，不是普通 `.agents/skills` 文件，但它在用户界面的交互感知上应与 skill 调用保持一致。

至少包括：

- 通过同一个 slash 资源面板被选择
- 选择后在输入区以与 skill 相同层级的已选资源表现呈现，而不是另起一套特殊外观
- 发送后进入与普通 skill 调用一致的执行中状态
- 完成后通过普通消息结果流回，而不是只弹一个独立提示或静默落盘

换句话说，`/init` 可以在数据模型上是 `builtin-command`，但在 UI 行为上应尽量复用 skill invocation 的感知路径。

### 对话调用效果

当对话作用域决定调用工作空间子 Agent 时，应满足以下语义：

1. 对话是主 Agent，负责理解用户目标、选择是否委派以及整合结果
2. 被选中的工作空间 Agent 作为子 Agent，在自己的工作空间根目录中执行任务
3. 子 Agent 的身份描述来自该工作空间 `AGENTS.md` frontmatter 中的 `name` 与 `description`
4. 子 Agent 执行结果回流给对话主 Agent，再由主 Agent 向用户输出最终答复
5. 如用户通过 `/` 显式指定工作空间 Agent，主 Agent 必须优先尊重该指定

此外，主 Agent 发起对子 Agent 的委派时，应该被建模为一次明确的 tool 调用，而不是普通消息中的隐式状态变化。

这意味着：

- 主 Agent 需要调用一个专门的“工作空间委派” tool
- 该 tool 产生独立的工具调用消息与渲染卡片
- 该卡片应展示目标工作空间 Agent、委派状态、结果摘要和可跳转入口
- 主 Agent 只能收到该次委派的最终结果摘要、失败原因或停止结论，不能收到子 Agent 的中间工具调用明细

这里的“子 Agent”必须是**真正独立的子 Agent 会话编排**，而不是仅把主 Agent 的当前工作目录切换到目标工作空间继续执行。

最小独立性要求：

- 子 Agent 拥有独立的 Agent 实例
- 子 Agent 拥有独立的消息历史与运行状态
- 子 Agent 在对应工作空间的 `cwd`、`AGENTS.md`、技能集合下构建系统上下文
- 主 Agent 只能读取子 Agent 的结构化最终结果或摘要回传，而不是直接复用同一条运行态消息流
- 子 Agent 的中间工具调用、审批过程与过程消息仅保留在子 Agent 自己的会话中

## 生成内容契约

`/init` 生成或改进出的 `AGENTS.md` 必须满足以下约束：

- 文件定位是“给在该工作空间中工作的智能 Agent 使用”
- 总长度控制在约 150 行，允许小幅浮动，但不得演变为冗长手册
- 必须保留并优化用户提供的整体结构，不改变核心语义
- 必须以 YAML frontmatter 开头，格式如下：

```yaml
---
name: 工作空间名称
description: 工作空间作用，以及何时应由对话功能调用该工作空间
---
```

- 必须包含以下正文内容块：
  - 工作空间名称
  - 工作空间作用
  - 工作空间拥有的技能
  - 必须文件索引
- frontmatter `description` 必须明确：
  - 何时触发
  - 触发效果
  - 这是对话功能调用该工作空间的主要触发机制

这里的 frontmatter `name` / `description` 需要同时满足两种用途：

- 对工作空间内的 Agent 提供本地协作说明
- 对对话主 Agent 提供“何时应该把任务委派给这个工作空间”的路由摘要

建议输出结构：

1. 标题与工作空间概览
2. 工作空间作用与适用任务
3. 技能与工具入口索引
4. 必须文件索引
5. 触发定义
6. 协作约束或补充说明

## 对已有 AGENTS.md 的处理

当根目录已存在 `AGENTS.md` 时，`/init` 不应简单覆盖。

推荐的改进策略：

- 保留现有仓库级硬约束，例如 workflow contract、路径约定、编码规范
- 删除明显重复、冗长或和“工作空间触发说明”无关的噪音表达
- 将现有说明重组为“工作空间初始化模板”需要的结构
- 如果已有文件已经覆盖 frontmatter `name` / `description`，则优先规范化表达，而不是重新发明含义
- 若已有 `AGENTS.md` 缺少可调度的 frontmatter `name` / `description`，则补齐到文件顶部
- 若已有正文中存在中文“名称” / “描述”，可用于辅助迁移，但不得替代 frontmatter 作为机器解析来源

这意味着 `/init` 更接近“结构化改写 + 增量补全”，而不是“无条件重置”。

这里的“改写”同样要求由 LLM 完成，而不是在代码里写死最终模板文本。

## 系统边界

### 为什么不是普通技能

普通技能的职责是向模型暴露可引用的说明文本；而本次能力需要：

- `/init` 只在工作空间可见
- 操作工作空间根文件
- 在首次没有本地技能的情况下也能工作
- 让对话把工作空间作为执行单元来委派，而不是只注入一段提示

因此它应在“资源选择 / slash 调用 / 工作空间服务 / Agent 委派”边界中单独建模。

### 与现有技能系统的关系

- `/init` 不替代 `.agents/skills` 的加载机制
- `/init` 生成的 `AGENTS.md` 可以与工作空间技能同时存在
- 生成内容可以引用工作空间中的技能目录和关键技能，但不应自动把 `/init` 本身写成普通技能
- 工作空间 Agent 的 frontmatter `name` / `description` 可以“像 skill 元信息一样被加载”，但其执行语义不是普通技能注入，而是子 Agent 委派

### 与对话作用域的关系

- `conversation` 工作空间本质上是无项目或轻项目上下文
- 因此 `/init` 不在对话中显示，也不允许通过对话上下文调用
- 但对话需要能够浏览并调用“其他项目工作空间 Agent”
- 若用户在错误上下文尝试执行 `/init`，系统应返回简短错误说明，提示其切换到具体工作空间

## 工作空间 Agent 注册模型

系统需要新增一类“工作空间 Agent 摘要”数据，其来源是工作空间根目录 `AGENTS.md` 的 YAML frontmatter。

建议字段包括：

- `workspaceId`
- `workspaceName`
- `agentName`
- `agentDescription`
- `workspacePath`
- `available`

解析规则建议：

- 必须优先读取文件顶部 YAML frontmatter 的 `name` 和 `description`
- 为兼容旧文件，可以把正文中的中文“名称” / “描述”作为迁移辅助信息，但不能把缺少 frontmatter 的工作空间标记为可自动调度
- 若缺少 frontmatter、`name` 或 `description`，则该工作空间不能作为可自动路由的子 Agent，只能提示用户先执行 `/init` 或补全 `AGENTS.md`
- `description` 应保持足够短，适合在对话页 `/` 资源面板中作为副标题展示，并适合注入主 Agent 的工作空间索引

## 自动调度模型

对话主 Agent 不应无条件为每个请求都调用子 Agent，而应按需判断。

推荐语义：

- 系统在构建对话上下文时，注入所有可用工作空间 Agent 的精简索引，只包含 `agentName`、`agentDescription`、`workspaceName`
- 主 Agent 先判断当前问题是否明显落在某个工作空间的职责范围内
- 若匹配明显，则可调用对应工作空间子 Agent
- 若没有明显匹配，则继续由主 Agent 直接回答，避免不必要的多 Agent 开销
- 若用户通过 `/` 显式指定，则跳过自动匹配，直接委派到指定工作空间 Agent

## 独立子 Agent 会话编排模型

本次需要把“主 Agent 调用工作空间”明确定义为一个独立的编排层，而不是简单参数切换。

建议编排结构：

- 主 Agent：对话页的会话 Agent，驻留在 `conversation` 作用域
- 子 Agent：某个工作空间的一次独立执行会话，驻留在该工作空间作用域
- 编排器：负责创建子 Agent、传入任务、等待结果、汇总回主 Agent

最小执行流程：

1. 主 Agent 决定是否委派
2. 编排器创建一个新的子 Agent 运行实例
3. 子 Agent 在目标工作空间路径下构建自己的系统 prompt、技能上下文和历史
4. 子 Agent 执行用户任务或主 Agent 转述任务
5. 子 Agent 返回结构化结果，例如总结、修改结论、产出说明、失败原因
6. 主 Agent 消费该结果并向用户生成最终回复

这意味着：

- 子 Agent 的生命周期独立于主 Agent 当前流式输出
- 子 Agent 可以有自己的工具调用、审批、失败、停止和摘要状态
- 主 Agent 与子 Agent 之间通过“任务输入 / 结果回传”通信，而不是共享同一份 `agent.state.messages`
- 主 Agent 接收的是最终结果层的数据，不接收子 Agent 中间工具调用事件流

### 子 Agent 会话与跳转

为了让用户真正感知到“这是一个独立子 Agent 会话”，系统需要把每次委派和对应的子 Agent 会话关联起来。

建议最小字段：

- `childSessionId`
- `childWorkspaceId`
- `childWorkspaceName`
- `childAgentName`

这些字段一方面用于编排器恢复和状态跟踪，另一方面用于 UI 跳转。

## 工具化委派模型

工作空间委派不应只是主进程内部的编排动作，它应暴露为一个专用 tool。

建议语义：

- tool 名称示例：`delegate_workspace_agent`
- 输入至少包含：
  - 目标工作空间 ID
  - 目标工作空间 Agent 名称
  - 主 Agent 转述的任务描述
- 输出至少包含：
  - 子 Agent 会话 ID
  - 目标工作空间信息
  - 执行状态
  - 结果摘要或失败原因

这样做的好处：

- 可以直接复用现有 tool renderer 体系
- 用户能在消息流里看到一次明确的委派行为
- 点击 tool 卡片即可进入对应子 Agent 会话

### UI 效果

工作空间委派 tool 应具有独立于普通文本消息的 UI 渲染效果。

至少包括：

- 显示这是一次“调用工作空间子 Agent”的行为
- 显示目标工作空间名称和 Agent 名称
- 在运行中显示进行态
- 完成后显示结果摘要
- 提供“查看子会话”或等价 CTA，点击后跳转到对应子 Agent 会话

该效果不应退化为：

- 只在普通 assistant 文本中写一句“我已调用某某工作空间”
- 只在内部状态中保留委派，消息流中没有独立可见卡片
- 把子 Agent 的每一步工具调用明细同步回主会话，导致主会话被过程噪音淹没

### 历史与状态边界

建议明确以下边界：

- 对话页的主 Agent 历史继续留在 conversation 会话里
- 子 Agent 的执行历史不应直接混入对话主会话原始消息表
- 如需持久化子 Agent 运行记录，应使用独立的工作空间执行记录或等价结构
- 主会话中默认只保留“委派发生了什么”和“子 Agent 返回了什么”的用户可理解摘要
- 子 Agent 的工具调用、审批细节和过程日志默认只保留在子 Agent 会话中，不回填主会话

### 停止与错误传播

- 当主 Agent 停止当前轮次时，编排器必须能同步中断正在运行的子 Agent
- 若子 Agent 失败，主 Agent 需要收到可解释的失败结果，而不是无声吞掉
- 若子 Agent 需要工具审批，审批流需要明确归属到当前委派任务，而不是伪装成主 Agent 自己的普通调用

## 子 Agent 会话导航模型

用户点击委派 tool 卡片后，系统应导航到该子 Agent 对应的会话视图。

建议导航行为：

- 若子 Agent 会话属于某个项目工作空间，则跳转到该工作空间会话详情页
- 跳转后用户可以看到该子 Agent 自己的完整消息历史、工具调用和运行结果
- 原对话主会话仍保留在 conversation 作用域中，不因跳转被销毁

这要求系统为“由对话发起的工作空间子会话”建立稳定的会话 ID 与路由映射。

## slash 显式指定模型

对话输入区应支持通过 `/` 显式指定某个工作空间子 Agent。

展示规则建议：

- slash 资源面板中新增 `工作空间 Agent` 分组或等价分组
- 每项展示 `agentName` 作为主标题，`agentDescription` 或 `workspaceName` 作为副信息
- 结果项可以采用与 skill 结果一致的行结构：左侧统一工作空间图标，中部 `name` / `description`，右侧显示工作空间名称或“工作空间”来源标签
- 选择后应插入结构化引用，而不是把文字当作普通正文

发送语义建议：

- renderer 将所选工作空间 Agent 作为结构化字段发送
- 主进程收到后，把它视为本轮显式委派目标
- 主 Agent 在本轮必须优先使用该目标，而不是重新做自动路由

## 实现形态建议

实现上可以引入“内置 slash 命令”与“工作空间 Agent 资源项”的抽象，至少满足：

- 能在 sender slash 资源面板中展示
- 能在主进程识别其执行语义
- 能依据工作空间或对话作用域进行过滤

建议数据字段包括：

- `/init`：
  - `id`: `builtin:init`
  - `name`: `/init`
  - `description`: “分析当前工作空间并创建或改进 AGENTS.md”
  - `scope`: `workspace`
  - `kind`: `builtin-command`

- 工作空间 Agent：
  - `id`: `workspace-agent:<workspaceId>`
  - `name`: 工作空间 Agent 的 `agentName`
  - `description`: `agentDescription`
  - `scope`: `conversation`
  - `kind`: `workspace-agent`

发送链路在识别到这些资源项时：

- `/init` 不走普通 `selectedSkills` 语义，而走专用执行分支
- `workspace-agent` 也不走普通 `selectedSkills` 语义，而走“显式子 Agent 委派”分支

但 UI 呈现上：

- `/init` 的选择态、发送态、完成态要尽量复用 skill 的表现层
- 不应让用户感知到它是一套完全不同的资源调用机制

## 执行流水

### `/init` 流水

1. 用户在工作空间会话中选择 `/init`
2. sender 将其作为一个结构化资源项发送给主进程
3. 主进程校验当前会话必须绑定项目工作空间
4. 主进程读取工作空间路径、已有 `AGENTS.md`、可见技能与关键文件索引
5. 主进程构造初始化提示并调用 Agent / LLM 执行
6. 执行结果写入 `<workspace>/AGENTS.md`
7. 主进程将生成结果作为普通消息流的一部分返回
8. UI 以与 skill 调用一致的结果路径反馈给用户，而不是仅使用旁路提示

### 对话委派流水

1. 用户进入对话作用域
2. 系统加载所有项目工作空间的工作空间 Agent 摘要
3. 用户发送消息时，先检查本轮是否显式指定了工作空间 Agent
4. 若存在显式指定，则直接作为委派目标
5. 若没有显式指定，则由对话主 Agent 基于工作空间 Agent 摘要决定是否委派
6. 主 Agent 调用“工作空间委派” tool
7. 编排器为被选中的工作空间创建独立子 Agent 运行实例
8. 子 Agent 在其工作空间路径下执行任务，并维护独立运行状态
9. tool 结果记录子 Agent 会话 ID、目标工作空间信息和摘要结果
10. 子 Agent 结果返回给主 Agent
11. 主 Agent 负责向用户给出最终结果，并保留跨工作空间的对话连续性
12. 用户点击 tool 卡片后，可跳转到对应子 Agent 会话

这里的“子 Agent 结果”仅指最终执行结果层的信息，例如：

- 完成摘要
- 修改产物说明
- 失败原因
- 停止结论

不包含以下内容：

- 中间工具调用列表
- 审批请求细节
- 过程中的逐步状态消息

## 验证

至少需要覆盖以下验证：

- 工作空间会话中可以发现 `/init`
- 对话作用域中不会发现 `/init`
- 无 `AGENTS.md` 时会创建文件
- 有 `AGENTS.md` 时会改进并保留关键仓库约束
- `/init` 的输出确实来自一次 LLM 调用
- 生成内容包含 frontmatter `name` / `description`、技能索引、文件索引和触发说明
- 生成内容长度大致受控在约 150 行
- 对话作用域能读取并展示工作空间 Agent 摘要
- 对话可按需自动调用工作空间子 Agent
- 对话可通过 slash 显式指定某个工作空间子 Agent
- 工作空间调用使用真正独立的子 Agent 会话编排，而不是只切换主 Agent 的工作目录
- 工作空间委派在消息流中表现为独立 tool UI
- 点击该 tool UI 后可以跳转到对应子 Agent 会话
- `/init` 的 UI 触发与结果表现与 skill 调用一致

如果涉及 UI 变更，仍需遵守仓库约束：相关 renderer 或 sender 设计工作应先对齐 OpenSpec，再遵循根目录 `DESIGN.md`。
