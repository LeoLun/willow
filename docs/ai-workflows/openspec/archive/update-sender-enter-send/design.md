# Design: update-sender-enter-send

## Overview

本次设计只解决一个明确问题：让聊天发送器在桌面工作台语境下具备更高频、稳定的键盘发送体验。

目标交互为：

- `Enter`：发送当前消息
- `Shift+Enter`：在编辑区插入换行

这组快捷键只在“普通输入态”成立；当技能选择面板或文件选择面板处于激活状态时，`Enter` 仍必须优先服务于面板确认，而不是直接发送消息。

## Current State

当前共享发送器位于 [`packages/sender/src/components/Sender.vue`](/Users/liujinglun/code/willow/packages/sender/src/components/Sender.vue)，已经具备这些输入相关能力：

- 中部使用 Tiptap 编辑器承载消息输入
- slash 技能面板与 `@` 文件面板共用一套触发与键盘导航管理
- `Enter` 在面板打开时已被用于确认当前高亮项
- 发送动作最终由 `handleSend()` 负责，内部仍包含空消息与未配置模型校验

当前缺失的是“普通输入态下的回车发送契约”。也就是说：

- 已有面板态键盘逻辑
- 已有显式发送逻辑
- 缺少把两者拼成稳定用户语义的规格约束

## Chosen Approach

采用“发送器统一分流、编辑器保留原生换行能力”的方案。

职责划分：

- `Sender`：负责决定当前按键是否应该走发送、面板确认或放行给编辑器
- `Editor`：负责处理真正的文本输入与换行渲染

这样可以避免把“发送动作”下沉到纯编辑器组件中，从而保持共享发送器作为业务交互边界的完整性。

## Interaction Contract

### Normal Input State

当输入器处于普通编辑状态，且没有技能/文件选择面板处于激活状态时：

- 用户按 `Enter`
- 系统阻止默认行为
- 若当前消息可发送，则直接触发与点击主发送按钮等价的发送动作
- 若当前消息为空白，则不发送，也不插入额外空行

### Explicit Newline State

当输入器处于普通编辑状态，用户按下 `Shift+Enter` 时：

- 系统不触发发送
- 编辑器保留换行行为
- 当前草稿、已选技能、已选文件和模型状态保持不变

### Picker Active State

当 slash 技能面板或 `@` 文件面板打开时：

- `ArrowUp` / `ArrowDown` 继续用于高亮项导航
- `Enter` 继续用于确认当前高亮项
- `Escape` 继续用于关闭当前面板
- 此时即使用户未按 `Shift`，`Enter` 也不得触发消息发送

### Streaming State

当发送器处于流式输出中时：

- 主按钮语义仍然是“停止”
- 本次变更不要求 `Enter` 在流式输出中承担停止快捷键
- 键盘发送语义仅约束“发送前输入态”

这样可以避免把“发送”与“停止”塞进同一个键盘动作，降低误触风险。

## Validation Rules

发送快捷键仍需复用当前发送链路已有的保护条件：

- 未配置可用模型时，不发送，并沿用当前引导用户前往设置的反馈
- 仅包含空白内容时，不发送
- 包含文件标签但正文为空时，仍按现有 `getTextWithoutFileTags()` 语义判定是否可发送

这意味着本次变更只改变“如何触发发送”，不改变“什么内容允许发送”。

## Component Boundary

建议实现边界如下：

- `Sender.vue`
  - 扩展当前 `handleEditorKeyDown()` 或等价逻辑
  - 在未打开面板时识别 `Enter` 与 `Shift+Enter`
  - 将普通 `Enter` 分流到 `handleSend()`
- `Editor.vue`
  - 保持作为 Tiptap 宿主
  - 仅在发送器要求拦截时阻止默认行为
  - 对 `Shift+Enter` 保持正常换行

不建议把发送判断完全放进 `Editor.vue`，否则编辑器会开始感知模型校验、流式状态和业务发送语义，边界会变差。

## Regression Risks

本次最主要的回归风险有三类：

### 面板交互回退

如果普通 `Enter` 发送逻辑优先级过高，可能导致 slash 技能选择和 `@` 文件选择在按回车时误发送消息，而不是确认当前选项。

### 多行输入回退

如果 `Shift+Enter` 没有被明确放行给编辑器，用户将失去多行输入能力。

### 空消息或异常状态误发送

如果新快捷键绕过现有 `handleSend()` 校验，可能导致空白消息、未配置模型状态或不完整编辑器状态被错误提交。

## Verification Focus

实现后至少要验证这些场景：

1. 普通文本非空时，按 `Enter` 直接发送。
2. 普通文本输入中，按 `Shift+Enter` 插入新行，且不会发送。
3. slash 技能面板打开时，按 `Enter` 选择高亮技能，不发送消息。
4. `@` 文件面板打开时，按 `Enter` 选择高亮文件，不发送消息。
5. 空白草稿按 `Enter` 不发送，也不污染输入区状态。
6. 点击右侧发送按钮的行为与 `Enter` 发送结果一致。
7. 发送成功后，输入区仍按现有语义清空。

## Out Of Scope

本次不定义以下内容：

- 为 macOS 或 Windows 单独设计不同发送快捷键
- 在 UI 中新增快捷键提示文案
- 为自动化 prompt 编辑器或其他非聊天输入框同步这套快捷键语义
