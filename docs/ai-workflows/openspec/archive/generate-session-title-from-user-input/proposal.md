# Generate Session Title From User Input

## Summary

将新会话标题生成从“等待首轮 AI 回复结束后生成”改为“用户提交首条消息后立即触发生成”。标题生成只基于用户首条输入内容，不再依赖助手首轮回复内容。

## Problem

当前会话标题生成发生在首轮 `agent_end` 之后，并且标题提示词同时引用用户首条消息和助手回复。这会导致会话在 AI 首轮生成期间持续显示空标题或“未命名会话”，尤其在长回复、工具调用、联网搜索或模型响应较慢时，侧边栏和历史列表缺少可扫描的会话标识。

代码现状是 [session.service.ts](/Users/liujinglun/code/willow/app/work/src/main/service/session.service.ts) 已经包含 `createSessionTitle(sessionId)`，并在 `sendMessage` 的订阅回调里等首轮 `agent_end` 后调用。也就是说，本次不是从零新增标题生成功能，而是调整既有标题生成代码的触发时机和输入来源。

用户的核心期望是：输入发出后，会话能尽快拥有一个可读标题，而不是等 AI 第一轮完整返回。标题用于快速识别会话主题，首条用户输入已经足够作为标题生成依据。

## Goals

- 用户在空会话中提交首条消息后，系统立即启动标题生成流程。
- 标题生成不等待首轮 AI 回复结束，也不读取助手首轮回复作为输入。
- 标题生成只在会话仍为空标题时写入，避免覆盖用户手动重命名或其他已生成标题。
- 标题生成失败时仍保留稳定回退标题，不影响主聊天发送流程。
- 渲染侧通过现有会话标题更新事件获得标题，不新增页面级交互负担。

## Non-Goals

- 不改变用户手动重命名会话的入口和优先级。
- 不新增独立标题编辑 UI。
- 不改变历史消息持久化格式。
- 不改变默认聊天模型、流式回复、工具调用或审批流程。
- 不要求为标题生成新增可配置模型设置。

## Approaches

### Approach A: 发送首条消息后立即基于用户输入生成标题

在发送消息流程确认这是空会话的首条用户输入后，异步启动标题生成。标题提示词只包含用户输入的可读文本，并继续复用现有标题清洗、空标题检查和 `SESSION_TITLE_UPDATED` 事件。

优点：

- 最符合“用户输入就直接生成”的体验要求。
- 不依赖 AI 首轮回复耗时，侧边栏更早出现标题。
- 复用既有 `createSessionTitle`、标题 Agent、标题清洗和事件同步逻辑，变更面集中在会话服务和标题提示词，风险可控。

缺点：

- 标题无法利用助手回复中的补充语义，个别标题可能不如首轮后生成精准。

### Approach B: 前端先用用户输入截断作为临时标题，后端首轮后再 AI 精修

用户提交后立即显示用户输入摘要，首轮 AI 回复结束后再用用户输入和助手回复生成更优标题。

优点：

- 首屏反馈最快，同时保留首轮后精修机会。

缺点：

- 标题会二次变化，可能打断用户对会话列表的空间记忆。
- 需要定义临时标题与正式标题的状态边界，复杂度更高。
- 与“不等待 AI 第一轮返回”的实现目标相比引入了额外行为。

### Approach C: 完全使用本地截断标题，不再调用标题生成模型

用户提交后直接用首条输入的前若干字符作为标题。

优点：

- 最快且无需模型调用。
- 离线或无 API key 时也稳定。

缺点：

- 标题质量较差，长 prompt、技能引用、文件引用或多段输入时可读性不足。
- 改变了现有“AI 生成短标题”的产品语义。

## Recommendation

采用 Approach A。它直接满足“用户输入就生成，不等待 AI 第一轮返回”的要求，同时保留现有 AI 标题生成质量、标题清洗、事件同步和手动重命名优先级。实现时只需把标题生成触发点前移到首条用户消息提交后，并让标题生成输入只包含用户首条输入。

## Success Criteria

- 新空会话发送首条消息后，标题生成在首轮 AI 回复完成前即可启动。
- 首轮 AI 回复很慢、执行工具或仍在流式输出时，会话标题仍可被生成并同步到侧边栏。
- 如果用户在标题生成完成前手动重命名，会话标题不会被自动生成结果覆盖。
- 如果标题模型失败或返回空内容，系统使用用户输入摘要或“新会话”作为回退。
- 后续非首条消息不会重复触发自动标题生成。
