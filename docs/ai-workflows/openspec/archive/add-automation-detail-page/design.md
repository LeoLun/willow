# Design: add-automation-detail-page

## Overview

这次改动是在现有自动化列表页之上补充“详情兼编辑”能力。目标不是重做自动化系统，而是把当前只适合浏览摘要、依赖弹窗编辑的列表页，扩展为一个更完整的工作台流程：

- 在 `/auto` 浏览自动化列表或空状态
- 点击列表项进入 `/auto/:automationId`
- 在详情页查看完整信息，并直接修改标题、提示词和关键配置后保存
- 继续从稳定的页面动作进入删除等后续操作

为了让“关键配置后保存”具备真实能力，本次设计补充一层最小自动化执行配置：

- 自动化本身持有可更新的 `workspaceId`
- 自动化本身可选持有 `modelId`
- 推理状态由所选模型配置自动推导，不作为自动化独立持久化字段

设计必须遵守 `DESIGN.md` 中的桌面工作台约束：清晰结构、紧凑密度、低装饰、稳定的主操作位置。

## Chosen Approach

采用“列表页 + 可编辑详情子路由”模式，而不是详情弹窗或一次性双栏重构。

原因：

- 最贴合用户提出的“点击自动化列表项进入”交互。
- 能自然使用面包屑表达 `Automations > 当前自动化名称`。
- 比弹窗更适合承载完整提示词和多组配置摘要，也更适合长表单编辑。
- 比双栏同屏改造更稳妥，避免这次 scope 膨胀到列表架构和响应式重构。

## Route And State Model

### Route Shape

- 保留 `/auto` 作为自动化列表页
- 新增 `/auto/:automationId` 作为自动化详情页

### Data Loading

- 列表页继续使用 `fetchAutomationList()`
- 详情页优先调用 `getAutomation(id)` 获取单条自动化完整数据
- 如果 store 中已有对应自动化摘要，可先用于页面骨架或标题占位，再等待详情数据完成
- 页面表单初始化基于详情数据快照，未保存编辑应与服务端原值区分
- 详情页还需要加载工作空间列表与模型列表，以支撑执行配置编辑和摘要展示

### Missing State

- 当路由 id 非法或数据不存在时，详情区展示“未找到该自动化”的受控空状态
- 空状态必须提供返回 `/auto` 的明确入口

## Page Structure

### Header

详情页头部使用轻量面包屑和页面动作区：

- 左侧：`Automations > 当前自动化标题`
- 右侧：保存作为主操作，取消或返回作为次操作；删除保持弱化或独立摆放

不要把所有状态标签、统计信息、筛选器堆到头部。头部只解决定位和主动作，详细信息下沉到正文与侧栏。

### Main Content

主内容区聚焦“这条自动化做什么”，并承担最主要的编辑表单：

- 标题字段：可编辑的自动化名称输入框，保持页面级标题层级
- 简短说明区：用于提示当前页面可修改并保存，不额外发明营销文案
- 正文内容区：可编辑的 prompt `Textarea`，保留换行，使用适合阅读和输入的内容容器
- 如果 prompt 较长，容器仍应保持稳定边距和良好可扫描性，不使用过强代码块视觉
- 需要明确脏状态、保存中和保存失败时的反馈位置

### Secondary Metadata Rail

右侧或次级区域承载“这条自动化怎么运行”，并允许用户修改关键配置：

- 状态
- 下次运行时间
- 上次运行时间
- 工作空间
- 重复规则
- 模型
- 推理能力摘要
- 运行环境 / 工作树信息（有值时展示真实值，无值时展示占位）

这一列应按分组组织，例如：

- `Scheduling`
- `Configuration`
- `Previous runs`

每组内部按字段性质混合使用 label/value 与紧凑表单控件展示，不要堆太多 Badge，也不要做成厚重设置面板。

其中：

- `工作空间` 与 `模型` 为可编辑字段
- `推理能力` 为只读摘要，来自当前选中模型；如果未选择模型，则回退到默认模型；如果默认模型也不存在，则展示明确占位

## Interaction Rules

### List To Detail Navigation

- 自动化列表项整体可点击
- 行内下拉菜单、编辑按钮、删除按钮等交互元素必须阻止冒泡，避免误跳转
- hover/active 态应让用户感知“可进入”，但仍保持工具型风格，不使用夸张动画

### Detail Actions

- 页面内直接编辑，不再把标题、prompt、状态、工作空间、重复、模型等主字段拆回独立编辑弹窗
- 删除动作可继续复用现有删除确认弹窗
- “测试”按钮若现阶段已有对应行为，可保留占位或现有触发语义；若实现尚未存在，本次规格仅要求动作位预留，不强制新增执行链路
- 保存动作需要覆盖成功、失败和无改动三种受控反馈

### Breadcrumb / Back Navigation

- 页面必须提供显式返回列表的路径
- 返回入口可由 breadcrumb 首段或单独次按钮承载
- 文案应直接表达导航目的，例如 `Automations`

## Data Mapping Guidance

基于更新后的共享类型，本次详情页最少应消费这些字段：

- `automation.title`
- `automation.prompt`
- `automation.status`
- `automation.trigger.summary`
- `automation.trigger.cronExpression`
- `automation.latestRun`
- `automation.workspaceId`
- `automation.modelId`

页面同时需要读取模型配置列表，以推导：

- 当前模型名称
- 当前模型是否支持推理
- 默认模型回退状态

执行语义规定如下：

- 若自动化保存了 `modelId`，执行时使用该模型
- 若自动化未保存 `modelId`，执行时继续使用当前默认模型
- `推理能力` 由最终解析到的模型配置的 `reasoning` 字段推导
- 若 `automation.modelId` 指向的模型已被删除，详情页必须展示明确的“模型不可用”状态，并允许用户重新选择有效模型后保存

可选字段若暂未持久化，也应在 UI 上稳定表现：

- next run：由现有调度信息或客户端格式化规则展示；无法精确计算时使用明确占位
- environment：如果当前数据模型没有独立字段，可先展示占位或从现有上下文推导的只读文本，但不能伪造不存在的持久化能力

## Visual Constraints

遵循 `DESIGN.md`，特别是：

- 使用稳定的页面头部 + 主内容 + 次级信息区层次
- 优先用 `Card`、`Separator`、轻边框做分区，而不是大阴影
- 页面主标题与正文保持高可读性，不做 Hero 式居中布局
- 整体密度偏紧凑，避免截图式设计里大面积无信息留白被直接照搬

参考图可作为信息编排方向，但实现时需要按仓库规范收敛：

- 保留“左主右辅”的信息架构
- 弱化纯视觉留白，把 prompt 内容区和元数据区做得更贴近业务工作台
- 保持主按钮层级唯一，不在头部同时制造多个同级高强调按钮

## Open Questions Resolved

### 是否需要新数据模型

需要最小增量扩展。为支持工作空间与模型的真实编辑能力，自动化需要补充最小执行配置：

- `workspaceId` 需要进入更新链路
- 自动化需要新增可选 `modelId`

但本次不新增自动化级独立 `reasoning` 字段。

### 是否必须实现完整执行历史

不必须。本次只要求最近执行摘要与明确的“无更多历史/从未运行”表达。

### 是否采用详情弹窗

不采用。详情视图需要稳定导航语义和更强的信息承载能力。

### 是否允许在详情页直接编辑

允许。本次详情页本身就是主要编辑载体，用户应能在同一路径内完成查看、修改和保存。

### 推理是否作为自动化独立字段保存

不作为独立字段保存。推理能力由所选模型配置自动推导和展示；如果后续确实需要独立推理策略，再单独起新的 OpenSpec change。
