# add-automation-detail-page

## Summary

为桌面端自动化能力补充“自动化详情页”，用户可从自动化列表项进入某条自动化的详情视图，并直接在该页面修改标题、提示词、状态、工作空间、重复规则和执行模型等关键配置；推理能力不再作为独立保存字段，而是随所选模型自动变化并展示，同时保留删除等现有管理能力。

## Problem

当前自动化页面只提供列表浏览和弹窗式编辑，用户只能看到压缩后的标题、工作空间和时间摘要，也无法在一个稳定页面里完整查看和修改配置。随着自动化数量增加，用户难以快速确认某条自动化“是什么、何时运行、最近有没有跑、当前配置是否正确”，并且对标题、提示词、工作空间、执行模型等字段的调整被迫挤在弹窗里完成，不利于长内容编辑与整体核对。

实现评估还暴露出一个结构缺口：当前自动化数据模型没有独立的模型配置字段，更新接口也不支持修改工作空间；同时“推理”在现有系统中属于模型配置属性，而不是自动化运行时的独立开关。如果不先补齐这层语义，详情页将无法对这些字段提供真实的保存能力。

## Goals

- 为每条自动化提供可进入的详情视图。
- 自动化列表项应具备明确的进入详情交互，而不是仅承担展示作用。
- 详情视图展示完整标题、提示词正文、状态、下次运行、上次运行和关键配置摘要。
- 用户可在详情页直接修改标题、提示词正文、状态、工作空间、重复规则和执行模型。
- 系统为自动化新增可持久化的执行配置，其中至少包括 `workspaceId` 和可选的 `modelId`。
- 详情页展示当前自动化的推理能力摘要，该摘要由所选模型配置自动推导，而不是单独保存一份“推理开关”。
- 详情页同时承担“查看 + 编辑 + 保存”的稳定工作台职责，而不是只读查看后再跳回弹窗。
- 详情布局遵循仓库根 `DESIGN.md`，保持桌面工作台、信息密度稳定、层级清晰。
- 详情页优先复用现有 `getAutomation` 与列表返回字段，只对自动化执行配置做最小增量扩展。

## Non-Goals

- 本期不新增自动化执行日志的完整历史列表、筛选或分页系统。
- 本期不引入独立于模型配置之外的自动化级“推理开关”或复杂执行策略编排。
- 本期不改变自动化调度逻辑、AI tools 能力或数据库主模型。
- 本期不实现多栏工作流编排、运行结果分析报表或可视化时间轴。

## Success Criteria

- 用户可以通过点击自动化列表项进入某条自动化详情。
- 在详情页中，用户无需进入编辑弹窗即可理解并修改该自动化的用途、运行频率、当前状态和关键配置。
- 自动化详情页与列表页之间的返回路径清晰，支持面包屑或同等级导航提示当前位置。
- 用户修改标题、提示词、状态、工作空间、重复或模型后，可以通过明确的保存动作提交变更，并获得稳定的成功或失败反馈。
- 自动化运行时会使用保存后的 `workspaceId` 和 `modelId`；当未设置 `modelId` 时，继续回退到当前默认模型。
- 详情页展示的推理状态会随着所选模型变化而更新，且不会伪装成一个当前系统并不存在的独立持久化字段。
- 页面结构与交互符合 `DESIGN.md` 的工作台式布局，不出现过度装饰、大面积无意义留白或多个竞争主操作。

## Viable Approaches

### Approach A: 保持详情页可编辑，但只扩展工作空间，不引入自动化级模型配置

详情页可直接编辑标题、提示词、状态、工作空间和重复规则；模型与推理继续只读展示，沿用全局默认模型。

优点：
- 改动最小。
- 不需要调整自动化执行链路。

缺点：
- 无法满足“这条自动化使用哪个模型执行”的核心查看与编辑诉求。
- 页面上展示的模型与实际运行语义仍然脱节。

### Approach B: 增加自动化级执行配置，持久化 `workspaceId + modelId`，推理状态随模型推导

保留 `/auto` 作为列表入口，新增 `/auto/:automationId` 详情路由；列表页点击后进入详情视图，详情页直接提供可编辑表单与保存动作。自动化实体新增可选 `modelId`，并允许更新 `workspaceId`；推理能力由所选模型配置自动决定并展示。

优点：
- 信息承载能力最好，符合桌面工具页结构。
- 路由稳定，后续扩展执行历史、更多配置块更自然。
- 与 `DESIGN.md` 的“列表页 / 设置页 / 详情区”模式一致。
- 用户可以一边查看最近运行和调度摘要，一边修改对应配置，心智更统一。
- 与现有 `SessionService.sendMessage(modelId)` 和模型配置体系天然兼容，扩展面最小。

缺点：
- 需要补自动化字段、共享类型、更新接口和详情数据映射。
- 需要把“推理”从独立编辑项调整为模型导出摘要，和最初口头描述略有差异。

### Approach C: 为自动化同时新增 `modelId` 与独立 `reasoning` 覆盖字段

自动化实体新增 `modelId` 与单独的 `reasoning` 布尔配置，页面允许同时选择模型并切换推理开关。

优点：
- 表面上最贴近“模型、推理都可编辑”的原始表述。
- 为未来更复杂的执行策略预留更大扩展空间。

缺点：
- 当前运行链路没有自动化级推理覆盖语义，需要额外定义“推理开关如何影响模型选择或调用参数”。
- 需要引入比当前需求更大的数据与运行时改造，scope 风险最高。
- 如果模型本身不支持推理，会引出额外的冲突校验和回退规则。

## Recommendation

采用 Approach B：在“可编辑详情子路由”的基础上，补一层最小可行的自动化执行配置。也就是：

- 自动化可保存自己的 `workspaceId`
- 自动化可选保存自己的 `modelId`
- 推理能力跟随所选模型自动显示，不新增独立 `reasoning` 持久化字段

这个方案既能满足详情页中最关键的执行配置编辑，又与当前模型配置和会话发送链路保持一致，避免为了一个尚未定义清楚的“推理开关”把 change 膨胀成更大范围的运行时重构。
