# 自动化规格

## 新增需求

### 需求：支持从列表进入自动化详情页

系统必须允许用户从自动化列表直接进入某条自动化的详情视图。

#### 场景：从列表项进入详情页

- **前提** 已存在一条或多条自动化
- **并且** 用户当前位于自动化管理页面
- **当** 用户点击列表项中除行内操作控件之外的区域
- **那么** 系统进入该自动化的详情视图
- **并且** 目标路由通过自动化 id 标识当前选中的自动化
- **并且** 编辑、复制、删除等行内操作不会误触发页面跳转

#### 场景：从详情页返回列表

- **前提** 用户当前正在查看某条自动化的详情页
- **当** 用户使用页面面包屑或“返回列表”入口
- **那么** 系统返回自动化列表页
- **并且** 用户能够清楚理解当前所处的页面层级

### 需求：提供自动化详情视图

系统必须为单条自动化提供独立的详情视图。

#### 场景：展示自动化核心信息

- **前提** 用户打开某条自动化详情页
- **当** 对应自动化存在
- **那么** 页面使用自动化标题作为主标题
- **并且** 页面以可阅读、可编辑的内容区展示完整提示词内容
- **并且** 页面展示自动化状态、下一次计划运行时间、最近一次运行时间以及关联工作空间

#### 场景：展示关键配置摘要

- **前提** 用户打开某条自动化详情页
- **当** 页面渲染辅助元信息
- **那么** 页面在信息可用时展示运行环境、项目或工作空间、重复规则、模型、推理能力摘要等简明配置摘要
- **并且** 对于没有持久化值的字段，页面使用明确占位，而不是静默省略

#### 场景：推理摘要随模型变化

- **前提** 用户打开某条自动化详情页
- **当** 页面渲染当前执行模型信息
- **那么** 页面根据当前选中的自动化模型或默认模型展示推理能力摘要
- **并且** 页面不会把推理能力伪装成当前系统中独立可保存的一项自动化字段

### 需求：支持在详情页修改自动化核心配置

系统必须允许用户在自动化详情页直接修改并保存核心配置。

#### 场景：修改标题与提示词

- **前提** 用户打开某条自动化详情页
- **当** 用户修改标题或提示词内容
- **那么** 页面将这些修改保留在当前编辑状态中
- **并且** 用户可通过明确的保存动作提交变更

#### 场景：修改状态与工作空间

- **前提** 用户打开某条自动化详情页
- **当** 用户修改状态或工作空间
- **那么** 页面使用合适的表单控件承载这些变更
- **并且** 保存后详情页展示最新配置结果

#### 场景：修改重复与模型

- **前提** 用户打开某条自动化详情页
- **当** 用户修改重复规则或执行模型
- **那么** 页面允许用户在侧边配置区完成编辑
- **并且** 页面保持主内容区与配置区的稳定工作台层级

#### 场景：修改模型后更新推理摘要

- **前提** 用户打开某条自动化详情页
- **当** 用户切换执行模型
- **那么** 页面中的推理能力摘要跟随最新模型变化
- **并且** 推理能力变化不要求用户再单独保存一个独立的推理开关

#### 场景：保存成功

- **前提** 用户在自动化详情页修改了一个或多个可编辑字段
- **当** 用户执行保存
- **那么** 系统提交这些变更并展示明确的成功反馈
- **并且** 页面不再把这些字段视为未保存状态

#### 场景：执行时使用自动化保存的模型

- **前提** 某条自动化已经保存了执行模型
- **当** 系统按计划运行该自动化
- **那么** 系统使用该自动化保存的模型配置执行提示词
- **并且** 当自动化没有保存执行模型时，系统回退到当前默认模型

#### 场景：保存失败

- **前提** 用户在自动化详情页修改了一个或多个可编辑字段
- **当** 用户执行保存但请求失败
- **那么** 页面展示明确的失败反馈
- **并且** 用户当前输入不会因失败被静默丢失

#### 场景：展示最近执行摘要

- **前提** 用户打开某条自动化详情页
- **当** 存在最近执行数据
- **那么** 页面在清晰标注的区块中展示最近一次执行摘要
- **并且** 页面能够区分“从未运行”和“暂无更多历史”

#### 场景：处理不存在的自动化 id

- **前提** 用户访问了一个不存在的自动化详情路由 id
- **当** 页面尝试加载该自动化
- **那么** 系统在内容区展示明确的未找到状态
- **并且** 页面提供返回自动化列表的入口

### 需求：自动化详情界面遵循项目设计规范

系统必须按照仓库根目录 `DESIGN.md` 的要求实现自动化详情页的 renderer 界面。

#### 场景：以工作台层级渲染详情页

- **前提** 用户打开自动化详情页
- **当** 详情界面被渲染
- **那么** 页面采用稳定的桌面工作台结构，包含清晰的头部、主内容区和次级元信息区
- **并且** 主内容区优先保证标题与提示词的可读性和可编辑性
- **并且** 次级区域按分组展示并承载状态和配置信息，而不是挤占页面头部

#### 场景：保持稳定的操作层级

- **前提** 用户打开自动化详情页
- **当** 页面动作区域被渲染
- **那么** 页面保持一个稳定的主操作区域，并将保存作为主要提交动作
- **并且** 危险操作在视觉上保持弱化或与普通操作分离
- **并且** 详情页不使用营销化 Hero 布局、过大的装饰元素或多个相互竞争的主按钮
