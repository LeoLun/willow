# 设计文档：新增独立无限对话模式

## 设计目标

这次变更不是新增一个“普通聊天页”，而是把“对话”提升成 Willow 的一级能力：

- 它不依赖项目目录。
- 它不暴露会话树。
- 它天然支持单条长期连续对话。
- 它的上下文压缩策略和悬浮球反馈都要与项目工作空间聊天区分开。

基于现有代码结构，推荐采用“系统级对话工作区 + 单条无限会话”的落地方式，对外呈现独立对话能力，对内复用现有会话基础设施。

## 架构决策

### 1. 引入系统级“对话作用域”

现有 `sessions.workspace_id` 为必填，因此直接复用工作区骨架最稳妥。

建议在工作区层引入 `kind` 或等价标识：

- `project`：现有项目工作空间。
- `conversation`：系统级对话工作区，仅允许存在一条。

该工作区的核心规则：

- 根目录固定为 `path.join(app.getPath("home"), ".willow")`。
- 首次进入对话功能时自动创建目录；目录不存在不应报错退出。
- 它不在“工作空间”分区中以普通项目身份出现。
- 删除项目工作空间时不得影响该系统级对话工作区。

这样做的原因：

- `AgentService` 仍可拿到一个明确的 `cwd`。
- 现有 Sender 的文件选择、技能发现、系统 prompt 中的 cwd 逻辑可继续工作。
- 数据层不需要额外复制一套 message / event / approval 流程。

### 2. “对话”只暴露一个无限会话入口

产品语义上，本次不是“新增一组全局历史会话”，而是“新增一个无限对话流”。

因此 renderer 不展示“对话历史树”，而是：

- 左侧栏在“工作空间”上方新增一个标题为“对话”的分区。
- 该分区只包含一个平铺入口，点击后进入 `/conversation` 或等价专用路由。
- 不显示折叠箭头、最近会话列表、“查看更多历史”或会话级下拉菜单。

主进程会话生命周期建议如下：

1. 进入对话页面时，查找系统级对话工作区。
2. 若不存在则创建工作区及其目录。
3. 查找该作用域下的“当前无限会话”。
4. 若不存在则自动创建一条会话并复用；之后持续追加消息。

这意味着：

- 对话模式对用户表现为“一条持续生长的聊天流”。
- 标题生成对该作用域不是必须能力；若保留，也只作内部兼容字段，不进入侧边栏 UI。

### 3. 路由与导航行为

建议新增专用路由，例如：

```text
/conversation
```

路由职责：

- 加载系统级对话工作区和当前无限会话。
- 复用现有消息区与输入区，但页面语义与项目工作空间页区分开。
- 页面头部和空状态文案遵循 `DESIGN.md` 的工作台风格，避免营销化描述。

导航规则：

- 左侧栏“对话”入口始终可用。
- “工作空间”区保留原有项目树与项目会话列表。
- 应用默认入口可优先落到 `/conversation`，或在已有“上次位置恢复”逻辑中把它作为无项目场景的默认兜底。

### 4. 对话模式使用独立压缩策略

现有 `context-compression` 适合“普通会话逐轮压缩”，但无限对话流还需要额外解决两个问题：

1. 历史摘要本身也会越来越长。
2. 用户偏好、长期约束、未闭环事项需要跨很多轮稳定保留。

因此对话模式不应只复用“单条摘要 + 最近 3-5 轮”的策略，而应采用分层滚动压缩：

#### 分层结构

- `长期事实层`：用户长期偏好、固定约束、已确认的工作方式。
- `未闭环事项层`：还在继续推进的问题、待办、待验证项。
- `滚动摘要层`：较早对话的阶段性摘要，按压缩检查点生成。
- `热窗口层`：最近 2-4 轮完整消息，直接传给主 Agent。

#### 检查点策略

当对话作用域接近上下文阈值时：

1. 保留热窗口不动。
2. 将热窗口之前、尚未压缩的完整轮次压缩成新的“检查点摘要”。
3. 从检查点中提炼出长期事实和未闭环事项，合并到稳定层。
4. 若“滚动摘要层”本身再次逼近预算，则把最早的若干检查点再压缩为更高层摘要，避免摘要无限增长。

#### 为什么必须独立

- 项目工作空间会话通常围绕单个项目展开，关键是“最近上下文 + 少量关键历史”。
- 无限对话流更像持续陪伴式对话，需要额外保留“用户偏好”“长期约束”“还没做完的事”。
- 如果继续沿用同一份摘要策略，摘要会越来越像普通流水账，长期稳定性不足。

### 5. 持久化建议

为了明确与普通项目会话压缩区分，建议为对话作用域引入独立的压缩状态结构，而不是仅复用 `session_context_summaries`：

- 可以新增 `conversation_context_snapshots`（或等价命名）存储检查点摘要。
- 可以新增 `conversation_context_state`（或等价命名）存储长期事实层、未闭环事项层、最后压缩游标。

最小字段建议：

- `session_id`
- `checkpoint_index`
- `summary`
- `stable_facts`
- `open_loops`
- `compressed_until_message_id`
- `estimated_tokens`
- `updated_at`

约束：

- 仅系统级对话作用域使用该结构。
- 删除普通项目会话不影响对话压缩状态。
- 若未来支持“重置对话”，应同时清理该作用域会话消息和压缩状态。

### 6. 事件模型需要携带“对话作用域”信息

悬浮球只应响应“对话”模式，因此当前只靠 `UPDATE_MESSAGE` 的 `agent_start/agent_end` 不够。

建议主进程在相关事件中补充至少一项：

- `chatScope: "conversation" | "workspace"`，或
- `workspaceKind: "conversation" | "project"`，或
- `isConversation: boolean`

要求：

- renderer 主聊天页可继续按 `sessionId` 匹配当前页面。
- 悬浮球渲染器必须额外判断该事件是否来自“对话”作用域。
- 若来自项目工作空间，即使正在流式输出，也不得进入动画状态。

### 7. 悬浮球动画门控

悬浮球行为调整为：

- 收到 `agent_start` 且事件属于“对话”作用域 → `isStreaming = true`
- 收到 `agent_end` 且事件属于同一对话作用域 → `isStreaming = false`
- 收到项目工作空间会话事件 → 完全忽略

这样可以保证：

- 悬浮球成为“对话功能活跃度”的视觉反馈，而不是“任意 AI 输出”的全局指示器。
- 项目内批量修改、自动化、工作空间问答不会误触发悬浮球动画。

### 8. UI 约束

涉及侧边栏和聊天页时，遵循 `DESIGN.md`：

- 使用现有 `@willow/shadcn` 侧边栏组件，不单独重造导航原语。
- “对话”分区应比“工作空间”更轻，不做树状层级，不增加装饰性视觉。
- 保持桌面工作台语义，文案直接、结构清晰、密度紧凑。

## 验证建议

### 主流程

- 首次进入“对话”页时自动创建 `~/.willow` 目录和系统级对话工作区。
- 未选择任何项目时，用户仍可直接发送消息并正常使用工具。
- 左侧栏稳定显示“对话”分区，且不存在历史对话下拉树。

### 压缩

- 无限对话达到阈值后触发独立压缩。
- 长期事实、未闭环事项、最近热窗口在压缩后仍能被模型读到。
- 多次压缩后，摘要自身不会无限膨胀到再次不可发送。

### 悬浮球

- 在“对话”模式发送流式消息时显示动画。
- 在项目工作空间会话中发送流式消息时不显示动画。
- 在“对话”模式结束流式输出后动画及时恢复为空闲状态。
