# 新增独立无限对话模式

## 动机

当前 Willow 的聊天入口完全绑定项目工作空间：

- 会话必须挂在某个 `workspaceId` 下面，用户无法直接开启一个“不属于任何项目”的长期对话。
- 左侧栏只有“工作空间”树，天然鼓励“项目 → 会话”的使用方式，不适合豆包式的无限对话流。
- 现有上下文压缩虽然已经支持长会话，但设计目标仍是“普通项目会话保留最近若干轮”，并没有针对单条无限对话流做分层压缩。
- 悬浮球当前监听 AI 输出事件即可展示动画，无法区分“项目会话正在执行”还是“对话功能正在对话”。

本次变更要把“对话”提升为一个独立于项目工作空间的产品能力：它有自己的入口、自己的上下文压缩策略、自己的悬浮球状态来源。

## 目标

1. 新增一个不与项目绑定的“对话”模式，默认根目录为用户主目录下的 `.willow`，例如 macOS 为 `~/.willow`。
2. 侧边栏在“工作空间”上方新增“对话”分区，作为独立入口；该分区不展示历史对话下拉列表。
3. 为“对话”模式设计独立的上下文压缩机制，适配无限对话流的长期连续使用。
4. 悬浮球动画只响应“对话”模式的流式输出；项目工作空间中的会话不再触发悬浮球动画。

## 范围

- 数据模型中引入“对话作用域”或等价的系统级对话工作区标识。
- 新增或改造对话入口路由、会话获取逻辑、侧边栏导航结构。
- 为对话模式定义独立的上下文压缩状态、触发策略与摘要结构。
- 调整悬浮球事件过滤逻辑，只消费“对话”模式的流式状态。

## 非范围

- 本次不提供“多条无限对话历史切换”能力。
- 本次不重做工作空间聊天页的交互结构。
- 本次不引入跨会话 RAG、知识库或长期记忆检索。
- 本次不改变自动化（Automation）与项目工作空间之间的绑定关系。

## 可选方案

### 方案 A：隐藏系统工作空间 + 单条无限会话（推荐）

思路：

- 在现有 `workspaces / sessions` 模型上增加“系统级对话工作区”概念。
- 该工作区根目录固定为 `~/.willow`，不在“工作空间”分组中以项目身份出现。
- “对话”入口始终指向这一个作用域下的当前无限会话。

优点：

- 最大限度复用现有 DAO、消息持久化、Sender、Agent 和事件体系。
- 能满足“非项目绑定”与“独立入口”的产品语义，同时控制实现风险。
- 更容易把对话模式的压缩机制和悬浮球过滤做成作用域分支，而不是全局重写。

缺点：

- 产品层面是独立“对话”，但底层仍沿用工作区/会话骨架，需要在实现中小心隐藏系统工作区细节。

### 方案 B：新增一套 Conversation 表与服务

思路：

- 在数据库中新增 `conversations`、`conversation_messages`、`conversation_context_*` 表。
- renderer、主进程、AgentService 分别走独立的 conversation 流程。

优点：

- 领域边界最清晰，“对话”与“项目工作空间”完全分离。
- 后续扩展多条对话历史、对话收藏、对话归档更自然。

缺点：

- 需要复制或重构大量现有会话能力，实施成本高，短期回报不成比例。
- 很容易在 Sender、消息流和工具审批上产生双份维护成本。

### 方案 C：统一抽象为 Thread 模型

思路：

- 彻底把 `session` 升级为 `thread`，支持 `conversation` / `workspace` / `automation` 等 scope。
- 所有聊天、压缩、事件、悬浮球都基于统一 thread scope 工作。

优点：

- 架构最完整，未来可扩展性最好。

缺点：

- 涉及面过大，超出本次“补齐无限对话能力”的合理范围。
- 与现有代码差异太大，不适合当前迭代节奏。

## 结论

采用 **方案 A：隐藏系统工作空间 + 单条无限会话**。

推荐理由：

- 它能完整满足这次的四个核心诉求，同时把改动控制在可落地的范围内。
- 现有 `SessionService`、`AgentService`、上下文压缩、悬浮球事件体系都可以复用，只需要按“对话作用域”做分支。
- 未来如果真的需要“多条全局对话历史”，仍可以在此基础上继续抽象，而不是一开始就投入 Thread 级大改造。

## 成功标准

- 用户在未选择任何项目时，仍可直接进入“对话”能力，并以 `~/.willow` 作为默认根目录。
- 左侧栏“对话”分区稳定显示在“工作空间”上方，且不再出现对话历史下拉树。
- 无限对话连续发送大量消息后，系统使用新的对话压缩机制维持可用上下文，不要求用户手动新建会话。
- 项目工作空间中的会话流式输出不再触发悬浮球动画；只有“对话”模式活跃流式输出时悬浮球才展示动画。
