# split-settings-page

## Summary

将当前单页设置改造为独立设置页面工作区：进入设置后不显示主应用左侧导航，只显示设置页专属侧边栏和右侧当前分区内容。设置侧边栏包含“返回应用”“外观”和“配置”；视觉与信息结构参考 [`ui/work.pen`](/Users/liujinglun/code/willow/ui/work.pen) 中的 `Willow / Settings / Appearance` 与 `Willow / Settings`，并继续遵守仓库根目录 [`DESIGN.md`](/Users/liujinglun/code/willow/DESIGN.md) 的桌面工作台规范。

## Problem

当前 [`Setting.vue`](/Users/liujinglun/code/willow/app/work/src/renderer/src/pages/setting/Setting.vue) 将外观、模型配置和 Tavily Key 管理放在同一个内容流里，用户进入设置后需要纵向扫描才能找到目标配置。随着配置项增多，这种单页结构会带来几个问题：

- 设置页面缺少专属信息架构，和主应用左侧导航混在一起后层级不够清晰。
- 外观设置和能力配置属于不同任务，却共享同一个页面标题与滚动区域。
- “设置”入口只能进入一个泛化页面，无法直接定位到“外观”或“配置”分区。
- 设计稿已经给出独立设置页结构，但当前实现尚未承接。
- 如果进入设置后仍保留主应用侧边栏，页面会变成“主应用侧边栏 + 设置侧边栏 + 内容区”的三栏结构；这不符合 `work.pen`，也会让用户误以为设置只是主应用里的嵌套面板。

## Goals

- 将设置从单一内容页拆分为独立设置工作区。
- 设置路由进入后隐藏或绕过主应用侧边栏，只呈现设置工作区自身的两栏结构。
- 设置页内部提供左侧设置导航，固定包含“返回应用”“外观”和“配置”。
- “外观”页承载主题模式设置，默认对齐设计稿 `Willow / Settings / Appearance`。
- “配置”页承载模型配置与 Tavily API Key 管理，默认对齐设计稿 `Willow / Settings`。
- 保留现有模型、Tavily Key、默认模型、主题模式等真实能力与交互语义。
- 提供“返回应用”入口，使用户能从独立设置页回到主应用上下文。

## Non-Goals

- 本次不新增新的配置类型、模型字段、Tavily 计费逻辑或后端 API。
- 本次不重构 `useConfigStore`、模型弹窗、Tavily Key 弹窗或主题持久化机制。
- 本次不重做主应用左侧导航的内容、工作空间列表或普通应用路由布局；但设置路由必须能脱离主应用侧边栏显示，以匹配设计稿。
- 本次不把工作空间 `AGENTS.md` 设置迁移到全局设置页。
- 本次不引入新的主题 token、品牌色或独立视觉系统。

## Success Criteria

- `/setting` 或等价设置入口进入独立设置工作区，并默认展示“外观”分区。
- 设置工作区不显示主应用左侧导航，不出现三栏布局。
- 设置工作区左侧设置导航显示“返回应用”“外观”“配置”，当前分区有明确选中态。
- 用户可在“外观”页切换主题模式，行为与现有设置页一致。
- 用户可在“配置”页完成现有模型与 Tavily Key 的新增、编辑、删除、设为默认等操作。
- 从聊天输入区“前往设置”、主侧边栏“设置”等现有入口进入设置页后，不丢失原有使用路径。
- 页面布局、密度、按钮层级、图标与分组方式符合 `DESIGN.md` 和参考设计稿的工作台风格。

## Viable Approaches

### Approach A: 单路由内用本地状态切换“外观 / 配置”

保持 `/setting` 一个路由，`Setting.vue` 内部新增设置侧边栏，并用本地状态决定右侧渲染外观或配置内容。

优点：

- 初期改动最小，不需要调整路由表。
- 可以最快复用当前 `Setting.vue` 里的业务逻辑。

缺点：

- URL 无法表达当前设置分区，后续无法直接跳转到配置页。
- 浏览器前进 / 后退和刷新后的分区恢复语义较弱。
- 随着设置项增加，单文件容易重新变成“大设置页”。

### Approach B: 独立设置壳层 + 子路由拆分

将 `/setting` 作为独立设置壳层，默认重定向到 `/setting/appearance`；新增 `/setting/appearance` 与 `/setting/configuration` 两个子路由。设置壳层负责左侧导航、返回入口和内容容器，子页面分别负责外观与配置内容。设置壳层必须在主应用侧边栏之外渲染，或通过等价机制在设置路由隐藏主应用侧边栏。

优点：

- URL 与信息架构一致，可直接定位到外观或配置。
- 设置壳层和具体设置页职责清晰，便于后续扩展更多分区。
- 更贴近参考设计稿中“设置独立页面 + 左侧二级导航 + 右侧内容页”的结构。
- 可以保留现有能力，同时减少单文件膨胀。
- 避免出现主应用侧边栏和设置侧边栏并列的三栏结构。

缺点：

- 需要调整路由表和现有设置入口。
- 需要从当前 `Setting.vue` 中拆出局部组件或页面，改动面略大。

### Approach C: 保持当前单页，只增加页内锚点导航

在当前设置页左侧或顶部增加锚点导航，点击后滚动到“外观”或“配置”区域。

优点：

- 对现有代码影响最小。
- 不需要改变数据加载和弹窗调用方式。

缺点：

- 不符合“设置页面单独拆分为独立页面”的目标。
- 外观和配置仍然共享一个滚动内容流，信息架构没有真正拆开。
- 与参考设计稿的独立设置工作区差异较大。

## Recommendation

采用 Approach B：独立设置壳层 + 子路由拆分。

这能最清楚地表达“外观”和“配置”是设置页内的两个独立页面，同时保留现有设置能力。实现时应优先复用当前 `Setting.vue` 的业务逻辑与现有 shadcn 组件，把改动集中在路由结构、页面拆分、主应用壳层集成和布局层级上，而不是扩展新配置能力。

## Clarification: 独立设置页不能显示主应用侧边栏

设置页必须严格参考 `work.pen` 的两栏结构：左侧是设置专属侧边栏，右侧是设置内容。当前实现额外保留主应用侧边栏并形成三栏布局，这是规格缺口。本变更明确要求实现修正：

- 设置路由渲染时，主应用侧边栏不应显示。
- 设置侧边栏的“返回应用”承担回到主应用的入口。
- 设计稿里的设置页左侧栏不是主应用左侧栏的补充，而是替代主应用侧边栏的设置专属导航。
