# Design: add-chat-skill-picker

## Overview

本次设计解决两个问题：

1. 聊天输入区从普通文本域升级为可扩展的 Tiptap 编辑器
2. 用户可以在当前消息级别显式选择技能，且技能覆盖全局与当前工作空间两个来源

实现时需要遵守仓库 [`DESIGN.md`](/Users/liujinglun/code/willow/DESIGN.md) 的工作台约束：输入区仍应是克制、紧凑、工具型界面，不做营销化富文本编辑器包装。

此外，本次还应参考用户提供的输入框示意图：整体是单个深色圆角容器，技能以顶部胶囊形式内嵌在容器上缘，底部控制条与右侧发送按钮共处同一输入器内部，而不是把技能选择、编辑区和操作区拆成多个松散块。

## Current State

### Renderer

- [`app/work/src/renderer/src/components/base/sender/index.vue`](/Users/liujinglun/code/willow/app/work/src/renderer/src/components/base/sender/index.vue) 仍使用 `InputGroupTextarea`
- 输入区已经承载发送、停止、模型切换、联网开关等关键交互
- `Plus` 按钮当前没有承接真实扩展能力

### Main / Core

- [`app/work/src/main/service/agent.service.ts`](/Users/liujinglun/code/willow/app/work/src/main/service/agent.service.ts) 创建默认 agent 时，会把工作空间路径作为 `cwd`
- [`packages/core/src/skills.ts`](/Users/liujinglun/code/willow/packages/core/src/skills.ts) 已能从全局技能目录和 `cwd/.willow/skills` 读取技能
- 但 [`app/work/src/main/service/session.service.ts`](/Users/liujinglun/code/willow/app/work/src/main/service/session.service.ts) 发送消息时仍只把 `data.message` 原样传给 agent

这意味着：

- “技能可发现”已经存在
- “技能可显式指定”尚未存在
- “当前工作空间技能”在底层其实可加载，但没有 UI 和发送协议来表达用户本轮想用哪个技能

## Chosen Approach

采用“前端结构化编辑 + 主进程技能发现与提示组装”的方案。

具体划分：

- renderer：Tiptap 编辑器、技能选择 UI、编辑器内容与技能标签的本地状态
- shared API：在 `SendMessageRequest` 中新增“选中技能”结构
- main：提供技能列表查询能力，并在发送消息时把显式选中技能组装为本轮输入上下文

## Skill Source Model

本次显式区分两类技能来源：

### 全局技能

来源于应用使用的全局技能目录，即当前 Electron 用户数据目录下的 `.willow/skills`。

这与 `@willow/core` 当前 `loadSkills()` 的 `userDir` 语义保持一致，避免产品层和底层目录定义不一致。

### 当前工作空间技能

来源于当前工作空间根目录下的 `.willow/skills`。

由于默认 agent 已把工作空间路径作为 `cwd`，该来源也与底层现有技能加载行为保持一致。

### 去重与优先级

技能列表展示和发送时都应遵循现有底层的同名冲突语义：

- 默认保留先进入优先级链的技能
- 当前实现中全局技能先于工作空间技能进入合并逻辑
- 如果未来希望改为“工作空间覆盖全局”，需要另起 OpenSpec 变更，不在本次范围内修改

为了避免 UI 与底层规则分裂，本次界面应展示最终可用技能集合，并标注技能来源，而不是让前端自行发明另一套覆盖逻辑。

## Input Editor Model

### Editor Structure

输入区改为 Tiptap 编辑器，但本次只开放最小必要能力：

- 顶部技能胶囊区
- 纯文本输入
- 多段落换行
- 技能标签展示与删除
- 底部控制条

不开放：

- 富文本样式按钮
- 标题、列表、图片、代码块等富文档能力

### Why Tiptap

选择 Tiptap 的原因不是“让输入区更花”，而是为了提供稳定的结构化载体，使后续这些能力可以自然扩展：

- 技能 mention / tag
- slash command
- 文件引用 token
- 受控序列化

## Interaction Design

### Skill Entry

技能入口应放在输入区现有的扩展位中，优先复用当前 `Plus` 按钮或与其同级的紧凑入口。

入口打开后，用户能看到：

- 全局技能分组
- 当前工作空间技能分组
- 每个技能的名称和简短描述
- 当前已选状态

除显式点击入口外，输入区还应支持 slash 搜索触发技能检索：

- 当用户在编辑区输入以 `/` 开头的查询时，系统进入技能搜索态
- 查询如 `/sp` 时，结果列表实时按技能名称和描述做匹配过滤
- 用户确认某一项后，该查询文本不作为普通消息正文保留，而应转为已选技能胶囊

### Search Result Layout

技能搜索结果列表应参考用户给出的第二张示意图，至少包含三段信息：

- 左侧：技能图标或统一技能标识
- 中部：技能名称与简短描述，描述允许截断
- 右侧末尾：来源标签，明确展示该技能来自 `全局` 还是 `工作空间`

来源标签要求：

- 位置稳定，靠近结果项最右侧
- 文案简短直接，不使用模糊缩写
- 在同名技能存在冲突语境时，来源标签必须足以帮助用户区分

### Reference Layout Mapping

参考图中的结构在本项目中映射为：

- 外层：单个圆角较大的输入容器，承载全部输入与控制元素
- 顶部：已选技能胶囊区，展示如 `Workflow Spec` 这类当前消息显式启用的技能
- 中部：Tiptap 编辑区，保留大面积可输入空间
- 底部左侧：扩展入口、模型选择，以及当前应用已存在的辅助控制
- 底部右侧：最明确的主发送按钮；若正在流式输出，则切换为停止语义

这意味着本次实现不应采用以下形态：

- 技能标签漂浮在输入框外部上方，像临时提示条
- 发送按钮脱离输入容器，单独悬浮在外
- 把模型选择和技能选择塞成多个相互独立的小卡片

### Selection Behavior

选择规则：

- 用户可为当前消息选择 0 个或多个技能
- 选中的技能以胶囊标签形式显示在输入容器顶部的受控区域
- 取消技能后，标签立即消失
- 发送成功后，本轮技能选择应清空，避免无意串到下一轮
- 当已选技能较多时，优先保持单行可扫读和横向换行秩序，而不是让标签挤压底部控制条
- 当用户通过 slash 搜索选中技能后，搜索面板立即关闭并回到正常输入态

### Empty States

需要覆盖：

- 没有全局技能
- 当前没有工作空间上下文
- 当前工作空间下没有技能
- 技能目录存在但无有效 `SKILL.md`

这些状态都应给出简短说明，但不阻断普通文本发送。

## Data Contracts

### Skill Summary For Renderer

需要新增面向 renderer 的技能摘要结构，至少包含：

- `name`
- `description`
- `filePath`
- `scope`，取值为 `global` 或 `workspace`
- `scopeLabel` 或等价派生字段，用于在结果项末尾稳定显示“全局”或“工作空间”
- 可选的失效 / 警告信息

该结构用于技能列表展示与发送时的精确标识。

### Send Message Payload

`SendMessageRequest` 需要扩展为：

- 原始编辑器纯文本消息
- 本轮选中的技能列表

选中技能不应只传名称，至少要传能稳定标识技能的字段，推荐包含：

- `name`
- `filePath`
- `scope`

这样主进程可以验证“用户选择的技能”是否仍然存在且属于当前可用集合。

## Main Process Behavior

### Skill Discovery API

主进程需要补充一个显式的技能发现接口，供 renderer 获取当前会话上下文下的技能集合。

建议行为：

- 当存在工作空间上下文时，同时返回全局技能和当前工作空间技能
- 当不存在工作空间上下文时，仅返回全局技能
- 返回结果尽量复用 `@willow/core` 的技能读取逻辑，而不是在 renderer 侧重新扫描文件系统

### Prompt Assembly

发送消息时，`SessionService` 不应把技能标签直接拼回编辑器文本，而应在主进程统一组装本轮 prompt。

推荐格式是一个受控前缀块，表达：

- 用户显式指定了哪些技能
- 每个技能的名称、描述、文件位置和来源
- 模型应优先按这些技能处理当前任务

正文消息仍保留用户原始输入，避免技能选择污染编辑器中的自然语言文本。

### Validation

主进程发送前需要重新校验：

- 技能是否仍在当前可用集合中
- 技能文件路径是否匹配当前枚举结果
- 工作空间技能是否确实来自当前会话工作空间

若校验失败：

- 忽略失效技能并记录可诊断信息
- 若仍有正文消息，则继续发送
- renderer 应获得可理解的失败或降级提示，而不是静默吞掉

## Renderer State Ownership

建议状态归属如下：

- `Sender`：编辑器实例、当前文本、已选技能、技能弹层开关
- 页面级聊天壳层：继续负责会话发送、停止与模型切换，不接管编辑器内部状态
- 独立 store 或 composable：技能列表请求与缓存，可按工作空间上下文刷新

这样可以避免：

- `Chat.vue` 过度膨胀
- 技能选择状态在多个页面间串联
- 输入区和发送协议耦合过深

## Visual Constraints

根据 [`DESIGN.md`](/Users/liujinglun/code/willow/DESIGN.md)，输入区改造必须满足：

- 输入器是一个完整的工作台容器，优先通过表面、边框和圆角建立整体感
- 编辑器仍是工作台型输入器，不做大块装饰性富文本工具栏
- 技能标签密度紧凑，可快速扫读，并与参考图一样具有明显但克制的胶囊识别性
- 技能选择弹层优先用 `shadcn-vue` 现有浮层、命令菜单或下拉能力组织
- slash 搜索结果项信息层级清晰，名称、描述、来源标签一眼可扫
- 主发送按钮仍保持最稳定的主操作位置
- 底部控制条应作为输入器内部的一条稳定操作带，而不是零散按钮堆叠
- 中部编辑区应保持足够留白，避免因技能区或底部控件导致输入空间过于压缩

## Resolved Decisions

### 技能是否自动长期绑定到会话

不绑定。本次选择只对当前发送的一条消息生效。

### 没有选技能时是否还能正常聊天

可以。显式技能选择是增强能力，不是发送前置条件。

### 是否修改底层技能目录约定

不修改。继续沿用 `@willow/core` 现有的全局目录与工作空间目录约定。

### 是否让 renderer 直接扫描文件系统

不建议。技能发现由主进程统一提供，避免前端重复实现文件扫描和覆盖逻辑。
