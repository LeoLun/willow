# add-chat-skill-picker

## Summary

为聊天输入区引入基于 Tiptap 的富文本编辑器，并让用户可以在发送单条消息前显式选择要启用的技能。技能来源覆盖：

- 全局技能目录
- 当前工作空间技能目录

本次变更聚焦“输入与发送链路”，目标是让用户既能继续像现在一样直接输入问题，也能在需要时明确指定本轮消息要优先使用哪些技能。

界面实现参考用户提供的输入框示意：采用一个整体圆角工作台输入容器，在顶部展示当前已选技能胶囊，底部保留扩展入口、模型相关选择和右侧主发送按钮。

## Problem

当前聊天输入区仍是普通 `textarea`，存在两个直接问题：

- 输入区无法承载更丰富的结构化交互，例如技能标签、命令片段或后续扩展型内容块
- 用户虽然理论上可以依赖模型自动匹配技能，但应用层没有提供“明确指定本轮使用哪些技能”的入口

同时，仓库底层已经具备技能加载能力：

- `@willow/core` 会从全局技能目录与当前 `cwd` 下的项目技能目录读取技能
- Electron 应用在创建 agent 时已把工作空间路径作为 `cwd`

但这些能力仍停留在底层 prompt 注入，用户在界面中看不到、也无法为当前一条消息做显式选择，导致“技能可用但不可控”。

## Goals

- 将聊天输入区升级为基于 Tiptap 的可扩展编辑器
- 保留现有发送、停止、模型切换、联网开关等输入区核心行为
- 在输入区内提供显式的技能选择能力
- 支持通过 slash 搜索快速检索技能，例如输入 `/sp` 时筛选匹配技能
- 技能来源同时覆盖全局技能和当前工作空间技能
- 让选中的技能只对当前发送消息生效，并通过发送链路可靠传递给 agent
- 让输入区视觉骨架贴近参考图中的一体式工作台输入器，而不是继续沿用普通表单 textarea

## Non-Goals

- 本次不实现完整的技能安装、发布、启停或编辑管理中心
- 本次不重做 `/skills` 页面为复杂后台
- 本次不引入附件上传、图片块、代码块等与技能无关的编辑器高级能力
- 本次不改变底层技能目录约定，只复用现有全局 / 工作空间技能发现语义

## Success Criteria

- 输入区从 `textarea` 升级为 Tiptap 编辑器，且原有消息发送流程保持可用
- 用户可在输入区看到并选择全局技能与当前工作空间技能
- 用户可通过 slash 搜索快速筛选技能结果，而不是只能在长列表中手动查找
- 每个技能结果项尾部都能明确看出其来源于全局还是当前工作空间
- 选中的技能在发送时能以结构化数据进入主进程，而不是仅停留在前端展示
- 主进程会把用户显式选择的技能转成稳定的本轮提示上下文，再发送给 agent
- 当工作空间没有技能、技能目录缺失或选中技能失效时，界面和发送链路都能受控降级
- 输入区形成“顶部技能胶囊 + 中部编辑区 + 底部控制条”的稳定三段式结构

## Viable Approaches

### Approach A: Tiptap + 技能标签节点 + 发送时显式注入技能上下文

在输入区中用 Tiptap 承载正文和技能标签，发送时把正文与选中技能分开提交给主进程，由主进程统一组装最终 prompt。

优点：

- 输入层和发送层职责清晰，后续扩展 slash command、文件引用更自然
- 技能是结构化数据，不依赖脆弱的字符串解析
- 最符合“指定技能”这一需求，而不是把选择仅做成视觉提示

缺点：

- 需要同时改动 renderer、shared API 和 main 进程发送链路
- 需要设计编辑器序列化与空内容校验规则

### Approach B: 保留纯文本消息语义，在 Tiptap 中插入类似 `@skill(name)` 的文本标记

仍使用 Tiptap，但技能只作为特殊文本 token 插入消息字符串，再由主进程或模型自行解析。

优点：

- 实现速度较快
- 对现有发送接口改动较少

缺点：

- 文本解析脆弱，重命名、转义和删除体验都容易出问题
- 前后端对“这是不是技能”很难保持严格一致

### Approach C: 保留现有输入框，在外部弹层或 `/skills` 页面里选择技能

不引入 Tiptap，只在输入区外增加技能选择器，把选中结果拼进普通文本。

优点：

- 改动面最小
- 学习成本最低

缺点：

- 无法完成“输入框接入 tiptap 组件”的目标
- 交互分裂，技能选择与输入内容脱节
- 难以支撑后续输入区结构化扩展

## Recommendation

采用 Approach A。它能同时满足“输入框改为 Tiptap”和“显式指定技能”的目标，并为后续输入区演进留下稳定结构。
