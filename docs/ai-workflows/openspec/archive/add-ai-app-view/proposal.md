# Proposal: AI 应用视图

## 概述

在聊天右侧边栏的 "应用" 标签页中，通过 `WebContentsView` + 自定义协议加载当前工作空间根目录下的 `index.html` 文件，使用户可以在 Willow 中直接运行和交互自定义 AI 应用。

## 动机

当前右侧边栏已有 `primary`（会话/工作空间信息）、`files`（文件树）和 `app`（占位文本 "目录的打开 index.html 文件"）三个标签页。`app` 标签页是预留的 AI 应用展示区域，但尚未实现。用户希望将自己的 HTML/JS 应用放在工作空间根目录，并在 Willow 中直接预览和交互。

当前聊天页在展开右侧应用区域后，还需要补齐桌面自适应规则：

- 中间对话区域需要最小宽度，避免右侧应用区域或窗口缩小时挤压核心聊天体验。
- 关闭左侧主导航时，中间对话区域不应因为释放出的空间继续变宽；释放空间应优先让右侧应用区域自适应变宽。
- 右侧应用区域需要在可用空间内自适应宽度，并继续保持用户拖拽宽度偏好与 WebContentsView bounds 同步。

## 约束

- 使用 Electron 原生 `WebContentsView`（非 `iframe`、非 `webview` 标签），保证完整的浏览器环境和安全性
- 通过自定义协议 `ai-app://` 加载文件，保证同源策略下的 `fetch` 能力
- WebContentsView 的大小和位置通过 DOM 锚点 + ResizeObserver + IPC 与渲染进程同步
- 协议注册必须在 `app.ready` 之前完成
- 工作空间根目录是动态的（用户可选择不同工作空间），协议处理器需支持动态路径映射

## 可行方案

### 方案 A: iframe 加载（不推荐）

在渲染进程中用 `<iframe>` 加载 `file://` 协议的文件。

- 优点: 实现简单，无需修改主进程
- 缺点: `file://` 协议下 `fetch` 受限，跨域问题严重，无法使用 `WebContentsView` 的安全隔离优势，Electron 社区不推荐

### 方案 B: WebContentsView + 自定义协议（推荐）

在主进程中注册 `ai-app` 自定义协议，创建 `WebContentsView` 并挂载到主窗口，渲染进程通过 DOM 锚点 + IPC 同步位置和大小。

- 优点: 完整浏览器环境，同源策略下 `fetch` 可用，安全隔离（contextIsolation），与 DOM 布局无缝集成，符合用户需求
- 缺点: 需要主进程协议注册（在 app ready 前）、IPC 通信、跨进程坐标同步

### 方案 C: 独立 BrowserWindow（不推荐）

为 AI 应用打开独立窗口。

- 优点: 天然隔离，实现简单
- 缺点: 与主窗口布局不集成，用户体验割裂，需额外窗口管理

### 自适应布局补充方案

#### 方案 D: 保持现有 flex 自然分配，仅设置中间区最小宽度（不推荐）

- 优点: 实现成本最低。
- 缺点: 左侧主导航关闭后，中间对话区仍会自然吃掉释放空间，无法满足“对话区不改变大小，右侧弹窗自适应宽度”。

#### 方案 E: 以中间对话区为稳定锚点，右侧应用区吸收释放空间（推荐）

- 优点: 聊天主体验稳定，左侧主导航开关不会造成对话内容横向跳动；右侧应用区域能获得更多展示空间，符合截图反馈。
- 缺点: 需要在 `Chat.vue` 中明确中间区最小宽度和右侧栏可用宽度计算。

#### 方案 F: 完全三栏可拖拽布局（暂不推荐）

- 优点: 灵活度最高。
- 缺点: 超出本次反馈范围，会引入新的布局状态、拖拽手柄和持久化规则，容易扩大改动面。

## 推荐方案

**方案 B + 方案 E**。使用 `WebContentsView` + 自定义协议实现 AI 应用视图，同时让聊天中栏成为稳定锚点，右侧应用区域在左侧主导航关闭或窗口空间变化时自适应宽度。

## 关键目标

1. 在 `app.ready` 前注册 `ai-app` 自定义协议（`standard`, `secure`, `supportFetchAPI`）
2. 创建 `AiAppViewService` 管理 `WebContentsView` 的完整生命周期（创建、加载、bounds 更新、销毁）
3. 在 `ChatRightSidebar.vue` 的 `app` 标签页中放置 DOM 锚点，通过 `ResizeObserver` + IPC 同步 bounds
4. 支持工作空间路径动态切换：切换工作空间时更新协议映射路径并重新加载
5. 协议 URL 格式：`ai-app://<relative-path>`，解析为 `<workspaceRoot>/<relative-path>`
6. 错误处理：`index.html` 不存在时显示友好的空状态提示
7. 聊天中间区域设置明确最小宽度，窗口空间不足时优先保护聊天可用性
8. 左侧主导航关闭时，中间对话区域宽度保持稳定，释放出的水平空间优先分配给右侧应用区域
9. 右侧应用区域宽度按可用空间自适应，并继续驱动 `WebContentsView` bounds 更新

## 非目标

- 不提供 AI 应用的编辑/热更新能力（用户手动刷新或重新打开标签页即可）
- 不处理 AI 应用内部的路由导航
- 不支持多窗口 AI 应用
- 不引入全局三栏拖拽布局系统
- 不改变聊天输入器、消息列表和左侧主导航自身的交互语义
