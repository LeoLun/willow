# Sender File Mention Spec

## ADDED Requirements

### Requirement: Support File Mention Trigger In Sender

`@willow/sender` SHALL support `@` as a file mention trigger inside the chat input editor.

#### Scenario: Open file picker from typed trigger

- **GIVEN** the sender receives file options from its host
- **WHEN** the user types `@` in the editor
- **THEN** the sender opens a file picker panel above the input container
- **AND** the panel uses the same placement and workbench visual density as the existing `/` picker

#### Scenario: Filter files by query

- **GIVEN** the file picker is open from an `@` trigger
- **WHEN** the user continues typing a query after `@`
- **THEN** the file picker filters options by file name and workspace-relative path
- **AND** the query does not mutate the host-provided file option data

#### Scenario: Open file picker from toolbar

- **GIVEN** the sender is focused or available
- **WHEN** the user activates the file reference toolbar control
- **THEN** the sender opens the same file picker with an empty query
- **AND** keeps editor focus available for continued input

### Requirement: Reuse Slash Picker Interaction Semantics

The file picker SHALL follow the same interaction semantics as the existing `/` skill picker.

#### Scenario: Navigate file picker with keyboard

- **GIVEN** the file picker is open and contains matching files
- **WHEN** the user presses ArrowDown or ArrowUp
- **THEN** the active file option changes within the visible result bounds
- **AND** the browser does not move the editor caret for that key press

#### Scenario: Select active file with Enter

- **GIVEN** the file picker is open and a file option is active
- **WHEN** the user presses Enter
- **THEN** the sender selects that file option
- **AND** removes the active `@query` trigger text from the editor
- **AND** inserts a file reference tag at the trigger position
- **AND** closes the file picker

#### Scenario: Close file picker with Escape

- **GIVEN** the file picker is open
- **WHEN** the user presses Escape
- **THEN** the sender closes the file picker
- **AND** resets active picker navigation state

### Requirement: Represent Selected Files As Structured References

The sender SHALL represent selected files as structured references instead of relying on plain text parsing.

#### Scenario: Insert selected file as editor tag

- **GIVEN** the user selects a file from the file picker
- **WHEN** the sender inserts the selected item into the editor
- **THEN** the inserted item is a compact file reference tag
- **AND** the tag stores at least `name`, `path`, `relativePath`, and optional `extension`
- **AND** the tag can coexist with normal text and skill tags in the same editor content

#### Scenario: Render file tag as markdown link text

- **GIVEN** the editor contains a file reference tag for `app.ts` at `/user/xxxx/app.ts`
- **WHEN** the sender derives plain message text from the editor
- **THEN** the file reference appears as `[app.ts](/user/xxxx/app.ts)`
- **AND** this mirrors the existing skill reference pattern such as `[$skillname](skill/xxx.md)`

#### Scenario: Avoid duplicate file references

- **GIVEN** a file is already referenced in the current editor content
- **WHEN** the user selects the same file again
- **THEN** the sender does not insert a duplicate file reference tag
- **AND** the picker closes consistently with existing duplicate skill selection behavior

#### Scenario: Extract selected files on send

- **GIVEN** the editor contains one or more file reference tags
- **WHEN** the user sends a message
- **THEN** the emitted send payload contains `selectedFiles`
- **AND** each selected file includes at least `name`, `path`, `relativePath`, and optional `extension`
- **AND** the emitted `message` text contains each file reference in `[文件名.后缀](文件地址)` format
- **AND** the host does not need to parse the plain message string to identify file references

### Requirement: Render File References In @willow/ui Messages

`@willow/ui` SHALL render file reference markdown generated by the sender as compact file tags.

#### Scenario: Render local file markdown as file tag

- **GIVEN** a message contains `[app.ts](/user/xxxx/app.ts)`
- **WHEN** `@willow/ui` renders that message through `MarkdownBlock`
- **THEN** the file reference is displayed as a compact file tag
- **AND** the tag shows `app.ts`
- **AND** the full file path remains available as title or equivalent accessible metadata
- **AND** the reference is not displayed as a normal external link

#### Scenario: Keep skill tag rendering unchanged

- **GIVEN** a message contains `[$workflow-spec](skill/workflow-spec.md)`
- **WHEN** `@willow/ui` renders that message through `MarkdownBlock`
- **THEN** the existing skill tag renderer still displays a skill tag
- **AND** the file reference renderer does not intercept this skill format

#### Scenario: Keep ordinary markdown links unchanged

- **GIVEN** a message contains `[OpenAI](https://openai.com)`
- **WHEN** `@willow/ui` renders that message through `MarkdownBlock`
- **THEN** the link is rendered by the ordinary markdown link renderer
- **AND** it is not displayed as a file tag

### Requirement: Keep File Data Host-Provided

`@willow/sender` SHALL receive file options from the host and SHALL NOT read workspace files directly.

#### Scenario: Sender receives file options through props

- **GIVEN** a host application renders `Sender`
- **WHEN** it wants to enable file mentions
- **THEN** it provides file options through sender props
- **AND** it may also provide loading and error states for file loading

#### Scenario: Sender remains independent from app/work runtime

- **GIVEN** `@willow/sender` is built or reused outside `app/work`
- **WHEN** its source dependencies are inspected
- **THEN** it does not import `app/work` source modules
- **AND** it does not call Electron IPC
- **AND** it does not depend on Pinia or Vue Router for file mention behavior

#### Scenario: app/work maps workspace files into sender options

- **GIVEN** `app/work` has a current `workspaceId`
- **WHEN** `SenderContainer` renders the shared sender
- **THEN** it loads workspace files through the existing host-side file API or composable
- **AND** maps only file nodes into sender file options
- **AND** passes loading and error states to the sender

### Requirement: Preserve Existing Sender Behaviors

Adding file mentions SHALL NOT regress existing sender behaviors.

#### Scenario: Slash skill picker still works

- **GIVEN** the file mention feature has been added
- **WHEN** the user types `/` and searches for a skill
- **THEN** the existing skill picker still opens, filters, navigates, selects, and inserts skill tags as before

#### Scenario: Existing send controls still work

- **GIVEN** the editor contains normal text, skills, files, or any combination of them
- **WHEN** the user sends a non-empty message
- **THEN** model selection, web search state, usage display, send action, and stop action continue to behave as before
- **AND** the editor clears after successful send

#### Scenario: Empty and error file states do not block normal sending

- **GIVEN** no files are available or file loading failed
- **WHEN** the user writes a normal text message
- **THEN** the user can still send the message
- **AND** the file picker displays a concise empty or error state when opened
