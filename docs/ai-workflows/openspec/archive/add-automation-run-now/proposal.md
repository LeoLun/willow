# add-automation-run-now

## Summary

在自动化详情页右上角新增“立即执行”按钮。用户点击后，系统应立刻触发当前自动化执行一次，不需要等待下一次 cron 计划点；触发成功后应立即进入本次执行创建的会话，并刷新左侧会话列表。

## Background

自动化能力已经支持创建、编辑、删除、定时调度、应用启动后的漏跑补偿，以及运行记录。当前详情页可以编辑自动化名称、提示词、工作空间、模型、状态和重复计划，但用户无法在保存配置后马上验证这条自动化是否能按预期执行，只能等待下一次定时触发。

主进程已有 `AutomationService.runAutomationById(...)` 作为统一执行入口，但 renderer / preload / IPC API 目前只暴露 CRUD。新增“立即执行”应该复用现有执行链路、运行记录和并发保护，而不是绕过调度服务另写发送逻辑。

实现验证中发现：如果立即执行 API 等待 `sessionService.sendMessage(...)` 完整完成后才返回，详情页按钮会长期停留在“执行中...”状态。用户更合理的期望是：点击后立刻进入自动化本次执行的会话，在聊天页观看 AI 继续流式执行，同时左侧会话列表出现该新会话。

## Goals

- 在自动化详情页右上角添加立即执行入口，点击后马上触发一次当前自动化。
- 新增 renderer 到主进程的手动执行 API，并让 store 能更新当前自动化的最新运行摘要。
- 手动执行复用现有自动化执行流程，包括工作空间上下文、模型选择、prompt 发送、运行记录、错误记录和并发保护。
- 手动执行成功启动后立即返回本次执行创建的会话，renderer 跳转到该会话。
- 手动执行成功启动后刷新左侧会话列表，使新会话出现在所属工作空间下。
- 手动执行不改变自动化的 cron 表达式、启停状态、下一次计划触发注册。
- UI 遵守仓库根 `DESIGN.md`，保持桌面工作台风格和紧凑操作层级。

## Non-Goals

- 不新增自动化运行历史列表或结果详情页。
- 不新增运行队列、重试策略、取消运行、并发优先级配置。
- 不改变 `automation_runs` 表结构，除非实现阶段发现现有 `run_kind` 类型无法兼容手动执行且必须迁移。
- 不支持从自动化列表页批量立即执行。
- 不把禁用自动化自动改为启用。

## Approaches

### Approach A: Renderer 直接复用 update 后等待定时触发

用户点击按钮时只保存当前表单并提示等待下一次调度。

优点：

- 实现量最小。

缺点：

- 不满足“点击后马上触发一次任务”的核心诉求。
- 用户仍然无法快速验证自动化。

### Approach B: 新增独立手动执行 IPC，复用 AutomationService 执行链路

新增 `RUN_AUTOMATION_NOW` 常量、请求/响应类型、controller、preload API 和 Pinia store action。controller 调用 AutomationService 的手动执行方法，服务层复用现有自动化执行语义，只对 run kind / scheduled time / 手动返回时机做适配。手动执行应在创建 session、写入 running run 并启动 AI 发送后立即返回 session，而不是等待整轮 AI 完成。

优点：

- 行为清晰，UI 与主进程边界稳定。
- 复用已有工作空间、模型、prompt、运行记录和并发保护。
- 后续若 AI tool 或列表页也需要手动执行，可以复用同一服务方法。
- 用户点击后可以马上进入会话观察执行过程，不会卡在自动化详情页按钮 loading。

缺点：

- 需要补齐 API、controller、preload、store 多层类型和注册。

### Approach C: 在 renderer 直接创建会话并发送 prompt

点击按钮时，renderer 从当前表单拿到工作空间、模型和 prompt，直接调用会话 API 发送消息。

优点：

- 表面上可以较快实现可见效果。

缺点：

- 绕开自动化运行记录、并发保护、错误记录和服务层校验。
- 使“自动化执行”的业务语义分裂到 renderer。
- 后续调试最新运行状态会不一致。

## Recommendation

采用 Approach B。手动执行是自动化执行能力的一种触发来源，而不是详情页的本地发送快捷方式。实现应新增显式 IPC/API，并让主进程服务层统一负责校验、运行记录和执行。

## Success Criteria

- 用户打开某条自动化详情页时，右上角可以看到“立即执行”按钮。
- 点击“立即执行”后，当前自动化马上开始一次执行，并使用已保存的自动化配置。
- 请求启动期间按钮进入禁用和加载状态，避免重复点击造成重复触发；启动成功后不继续停留在详情页等待 AI 完成。
- 启动成功后，应用跳转到本次执行创建的会话页面。
- 启动成功后，左侧会话列表刷新并展示本次执行创建的会话。
- 启动成功后页面展示“已开始执行”一类成功反馈，并让本次运行在自动化最新运行摘要中体现 running/manual 状态。
- 启动失败时页面展示错误反馈；启动后后台执行失败时，运行记录保留失败状态和错误信息。
- 自动化原有定时调度、保存、重置、删除行为不被破坏。
