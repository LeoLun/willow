## 背景

当前应用已经具备工作空间、会话、模型配置和主进程 Agent 执行链路，但自动化页面仍是空壳，系统也没有“按触发器自动向某个工作空间发起任务”的基础设施。`app/work/src/main/service/session.service.ts` 已经封装了会话创建、消息发送和活动流状态维护，`app/work/src/main/controllers/init.controller.ts` 也提供了应用启动时初始化数据库和默认工作空间的入口，这为自动化调度和补偿执行提供了现成挂点。

本次需求同时涉及数据库结构、主进程调度、IPC 契约和 renderer 页面，且明确要求本期只支持定时触发，但需要为未来邮件等触发器预留扩展空间，因此需要先把数据模型和执行边界设计稳定。

## 目标 / 非目标

**目标:**

- 新增自动化记录模型，保存工作空间、提示词、触发器配置、执行模式和执行状态。
- 提供可扩展的触发器抽象，本期实现定时触发器。
- 支持自动化列表页、空态引导、右上角“添加自动化”入口，以及创建自动化表单。
- 支持自动化执行时在“新建会话”与“复用固定会话”两种模式中选择，默认选择“新建会话”。
- 在应用进程重启后补偿最近一次错过的计划时间；如果错过多次，只补最近一次。
- 页面交互优先复用仓库内已有的 shadcn-vue 组件体系。

**非目标:**

- 本期不实现邮件、Webhook 或文件系统事件等其他触发器。
- 本期不实现复杂的自动化编排、条件判断、串并行步骤或多动作流程。
- 本期不实现自动化执行历史的完整独立页面，只保留支撑列表展示和补偿判断所需的最小执行记录。
- 本期不改造底层 Agent 能力本身，只复用现有会话发送链路发起自动化执行。

## 设计决策

### 决策：用独立自动化实体加“触发器类型 + 触发器配置”建模，而不是为定时任务单独硬编码表结构

数据库新增 `automations` 主表，承载自动化通用字段：名称、工作空间 ID、自定义提示词、是否启用、触发器类型、触发器配置、执行模式、绑定会话 ID、最近计划执行时间、最近实际执行时间、最近执行状态等。`triggerType` 使用字符串枚举，`triggerConfig` 使用 JSON 文本保存具体触发器配置；本期定时任务只写入一种定时配置结构，但表结构不把字段拆死在 cron 专用列上。

这样做的核心目的是把“是什么触发”与“自动化执行什么”拆开。未来新增邮件等触发器时，可以在不破坏已有记录结构的前提下，增加新的 `triggerType` 和配置解析器，而不需要对表结构做大幅回滚式调整。

备选方案：

- 为本期只建 `cron_expression`、`timezone` 这类定时专用列。优点是简单；缺点是未来引入非定时触发器时会迅速失去可扩展性。
- 把触发器单独拆成一张 `automation_triggers` 表。优点是更规范；缺点是当前一条自动化只需要一个触发器，首版会引入不必要复杂度。

选择单表 + 可扩展配置方案，因为它最符合当前需求边界，且仍保留清晰升级路径。

### 决策：执行链路复用现有 SessionService，而不是为自动化单独实现一套 Agent 运行栈

自动化触发时，主进程中的 `AutomationRunner` 不直接操作底层 Agent，而是复用 `SessionService` 现有能力完成“确保目标会话存在 -> 组装消息 -> 发送执行”的链路。对于“新建会话”模式，运行前创建一个新会话；对于“复用固定会话”模式，首次执行时创建并保存绑定会话，后续持续向该会话追加上下文。

这让自动化与手动会话共享同一套消息持久化、工具审批、事件分发和模型配置逻辑，避免出现两套执行栈长期分叉的问题。自动化只是新增了一种“消息是谁发起的”入口，而不是另一种完全独立的运行时。

备选方案：

- 为自动化专门创建无会话执行通道，仅保存结果摘要。优点是链路短；缺点是会绕开现有消息历史、工具审批和 UI 兼容能力。
- 总是新建会话，不支持固定会话复用。优点是实现简单；缺点是不满足用户要求，也无法支持持续上下文的自动化场景。

选择复用 `SessionService`，因为这是最稳妥、最贴合现有代码结构的方案。

### 决策：新增独立调度服务，基于 `node-cron` 注册标准 cron 任务并执行最近一次遗漏补偿

主进程新增 `AutomationSchedulerService`，职责分为三块：

- 启动时读取所有启用中的自动化并恢复调度。
- 在自动化新增、修改、启用、停用、删除后同步刷新对应 `node-cron` 任务。
- 在应用启动恢复阶段检查最近应执行时间，若存在遗漏，只补最近一次计划时间。

定时触发不再自定义内存计时器计算下一次触发时间，而是统一使用 `node-cron` 注册任务。自动化记录中的 `triggerConfig` 对于定时任务至少保存 `cronExpression`，并使用标准五段 cron 格式（分钟、小时、日、月、星期）定义计划；如实现中需要时区支持，可额外保存 `timezone` 并交由 `node-cron` 处理。创建和编辑时应先通过 `node-cron` 的校验能力或同等规则校验表达式有效性，再允许保存。

为了兼顾易用性与标准化，renderer 表单不直接把所有用户都暴露给原始 cron 字符串，而是提供四种调度模式：

- 每天：用户选择一个时间，例如 `09:00`，转换为 `0 9 * * *`
- 每小时：无需额外时间输入，转换为 `0 * * * *`
- 每周：用户选择星期几和时间，例如周一 `09:00`，转换为 `0 9 * * 1`
- 自定义：直接输入 cron 表达式，并按标准格式校验

持久化时统一写入 `triggerConfig.cronExpression`，必要时可额外保留 `schedulePreset`、`weekday`、`time` 这类纯 UI 辅助字段，但调度系统始终只依赖标准 cron 表达式作为最终执行依据，避免出现 UI 状态和调度状态不一致。

补偿算法采用“按计划时间计算最近应执行点”的方式，而不是按进程下线时长逐条回放。对于某条自动化，如果当前时间与 `lastScheduledAt` 之间错过多次计划点，只取最近一个计划点作为补偿执行目标，并在执行后将该次计划时间记为已处理。这样既满足“只补最近一次”的业务要求，也避免应用重启后瞬间触发大量历史补跑。

备选方案：

- 只依赖进程内调度，不做重启补偿。优点是开发快；缺点是不满足核心需求。
- 将所有错过的时间点全部补跑。优点是语义完整；缺点是与本期规则冲突，也可能导致重启后过量执行。

选择独立调度服务，是为了把“记录管理”和“定时执行”职责隔离开，同时保证启动恢复逻辑有明确归属。

### 决策：用“自动化记录状态 + 最近一次执行摘要”表达列表信息，不在首版引入完整 execution history 页面

列表页首版只需要帮助用户理解“这条自动化是什么、绑定哪个工作空间、怎样触发、当前是否启用、最近执行是否成功”。因此 `automations` 主表中直接保留最近一次执行摘要字段，例如 `lastRunStatus`、`lastRunAt`、`lastErrorMessage`、`lastScheduledAt`。如果实现中发现需要更稳的补偿计算和故障排查，可以额外补一张轻量 `automation_runs` 表记录执行事件，但 renderer 首版不消费复杂历史。

这样能把页面复杂度控制在需求范围内：列表、空态、创建入口和基础状态足够支撑第一期；后续如需执行日志，再在此基础上扩展详情页和历史记录展示。

备选方案：

- 首版就做完整执行日志页面。优点是信息全；缺点是超出当前需求，会拖慢交付。
- 完全不记录最近一次执行摘要。优点是表结构最小；缺点是列表页缺乏可观察性，也不利于排查补偿执行。

选择“最近一次摘要优先”，因为它兼顾可见性与实现成本。

### 决策：自动化页面采用“列表页 + 表单对话框/抽屉”的轻量交互，而不是新增复杂多步向导

`/auto` 页面保留为自动化列表页。页面右上角提供“添加自动化”按钮；空列表时显示引导卡片和同一入口。创建/编辑自动化使用 shadcn-vue 组件组合出单屏表单，字段至少包括：名称、工作空间、提示词、触发方式、定时配置、执行模式、启用状态。执行模式默认选中“创建新会话”，同时允许切到“复用固定会话”。

这类信息密度更适合单页表单而非多步向导，因为字段数量有限，且用户需要在一个视图里同时核对工作空间、提示词和定时配置。使用对话框或抽屉可以减少路由切换，也更贴合现有页面规模。

备选方案：

- 独立创建页。优点是空间更大；缺点是对当前体量偏重。
- 多步 wizard。优点是步骤清晰；缺点是字段并不多，反而增加操作负担。

选择列表页 + 轻量表单交互，以最小改动满足需求。

### 决策：AI 创建自动化先生成预览草稿，再由用户通过现有表单确认落库

面向 AI 的自动化创建能力不直接调用现有 `createAutomation` 持久化真实记录，而是新增一个“创建草稿”工具结果。Agent 在理解用户意图后，返回结构化草稿数据，例如名称、提示词、工作空间提示、执行模式、调度配置和预览摘要；聊天区把这类结果渲染为专用预览卡片，展示类似“每天 9:00”“论文晨间提醒”这类摘要信息，并附带“创建”按钮。

用户点击卡片后，renderer 复用现有自动化表单组件，将草稿字段作为初始值带入；用户可以继续修改名称、工作空间、提示词和调度方式，最终提交时仍然通过既有 `createAutomation` IPC 接口和表单校验链路落库。这样可以把 AI 的职责限制在“帮助理解并准备配置”，把最终确认留给用户，同时避免为了 AI 创建再复制一套后端校验和写库入口。

备选方案：

- 让 AI 直接调用真实创建工具并借助审批弹窗确认。优点是工具模型简单；缺点是审批弹窗不适合承载完整表单编辑，也无法满足“先预览再修改”的交互。
- 不做正式工具，只让 AI 输出结构化文本，由 renderer 自行解析消息生成卡片。优点是接入快；缺点是协议脆弱，后续扩展到更新、删除和查询时难以维护。

选择“草稿预览 + 表单确认”，因为它最符合用户预期，并且能最大程度复用已有自动化表单与后端校验。

### 决策：面向 AI 的自动化管理采用独立工具集，并与现有审批能力协同

在现有统一工具体系上新增自动化相关工具，覆盖查询列表、查看详情、创建草稿、更新自动化和删除自动化。`list` 与 `get` 属于低风险读取工具，可直接执行；`update` 与 `delete` 属于会改变用户配置的写操作，继续复用已有工具审批能力，在真正执行前等待用户批准。`create` 则不直接暴露为真实写库工具，而是通过创建草稿工具回到上一个决策。

工具的输出不只面向模型，也需要被 renderer 稳定消费，因此创建草稿工具必须返回明确的结构化字段，而不是自由文本摘要。聊天消息渲染层需要识别该工具结果并映射为自动化预览卡片；若缺少必要字段，则退回普通工具结果展示，避免破坏通用工具消息渲染。

选择独立工具集，而不是让自动化能力散落在现有会话外部命令里，是为了让 Agent 能以一致方式管理自动化，同时与已有工具审批、工具消息展示和调用链路保持统一。

## 架构与数据流

### 主进程模块划分

- `AutomationDao`：读写自动化记录，封装查询启用中自动化、按 ID 查询、更新最近执行状态等数据库操作。
- `AutomationService`：承接 IPC 请求，负责创建、更新、删除、启停自动化，以及参数校验和 DTO 映射。
- `AutomationSchedulerService`：管理已注册的 `node-cron` 任务，负责启动恢复、重建调度、停用取消、遗漏补偿判定。
- `AutomationRunner`：执行单次自动化任务，负责解析执行模式、准备会话、调用 `SessionService.sendMessage`，并记录执行结果。
- `AutomationToolService`（命名可在实现时调整）：为 AI 工具提供自动化列表、详情、草稿生成、更新和删除所需的查询与映射能力，内部复用现有 `AutomationService` / DAO。
- 聊天工具结果 renderer：识别自动化草稿工具结果并渲染预览卡片，处理“创建”按钮点击后打开自动化表单。

### 触发执行链路

1. 用户在 renderer 创建或编辑自动化。
2. renderer 通过 preload / IPC 调用主进程接口保存记录。
3. `AutomationService` 写库后通知 `AutomationSchedulerService` 为该自动化建立或刷新 `node-cron` 调度任务。
4. 到达计划时间后，`AutomationSchedulerService` 调用 `AutomationRunner.run(automation, scheduledAt)`。
5. `AutomationRunner` 根据执行模式：
   - 新建会话：创建新 session，并用自动化提示词发起执行。
   - 复用固定会话：查找 `boundSessionId`；不存在则先创建并回写，再发送消息。
6. 运行结果回写到自动化记录的最近执行摘要字段；失败时保存错误摘要，供列表展示。

### AI 创建预览链路

1. 用户在聊天中提出“帮我创建一个每天 9 点提醒写论文的任务”之类的请求。
2. Agent 调用自动化创建草稿工具，返回结构化草稿，包括预览标题、时间摘要、默认提示词、推荐工作空间和标准化触发配置。
3. renderer 将该工具结果渲染为预览卡片，而不是直接显示原始 JSON。
4. 用户点击卡片上的“创建”按钮。
5. 自动化表单对话框打开，并将草稿字段作为默认值带入。
6. 用户确认提交后，仍然通过现有自动化创建接口落库并刷新列表/调度。

### 启动补偿链路

1. 应用启动时，`InitController.init()` 完成数据库初始化后，追加自动化调度恢复步骤。
2. `AutomationSchedulerService` 拉取所有启用中的自动化。
3. 对每条自动化，根据其触发器配置、`lastScheduledAt` 和当前时间计算“最近一个本应触发但尚未处理的计划时间”。
4. 若存在该时间点，则立即触发一次补偿执行，并将这次执行标记为对应的 `scheduledAt`。
5. 无论是否触发补偿，都重新注册该自动化对应的 `node-cron` 任务。

## 数据模型

建议在 `app/work/src/main/db/schema.ts` 中新增 `automations` 表，字段方向如下：

- `id`
- `name`
- `workspaceId`
- `prompt`
- `triggerType`
- `triggerConfig`：定时任务首版至少包含 `cronExpression`，使用标准五段 cron 表达式；前端可按需附带 `schedulePreset`、`weekday`、`time` 等辅助字段；可选包含 `timezone`
- `runMode`：`new_session` | `reuse_session`
- `boundSessionId`：仅在复用模式下使用
- `enabled`
- `lastScheduledAt`
- `lastRunAt`
- `lastRunStatus`：`idle` | `success` | `failed`
- `lastErrorMessage`
- `createdAt`
- `updatedAt`

如果实现补偿逻辑时发现单表不足以表达“某次计划时间是否已处理”，可以补充轻量 `automation_runs` 表，至少记录 `automationId`、`scheduledAt`、`startedAt`、`finishedAt`、`status`、`errorMessage`。但这张表在首版只作为内部执行记录使用，不强制暴露到独立 UI。

## 前端设计

### 页面结构

`app/work/src/renderer/src/pages/auto/Auto.vue` 改造为真正的自动化管理页，包含：

- 顶部标题与描述。
- 右上角“添加自动化”按钮。
- 空态卡片：列表为空时说明自动化用途，并提供主操作按钮。
- 自动化列表：每项展示名称、工作空间、触发方式、执行模式、启用状态和最近执行摘要。

### 表单交互

表单建议使用现有 shadcn-vue 组件组合：

- `Dialog` 或 `Sheet` 承载创建/编辑面板。
- `Input` 填写名称。
- `Select` 选择工作空间。
- `Textarea` 输入提示词。
- `RadioGroup` 选择执行模式，默认 `new_session`。
- 定时配置区域提供四种模式切换：
- “每天”：显示时间选择器。
- “每小时”：仅显示模式选择，不再要求输入时间。
- “每周”：显示星期选择器与时间选择器。
- “自定义”：显示 cron 表达式输入框。
- 页面提交前将预设模式转换为标准 cron 表达式；编辑已有自动化时，如果表达式可识别为预设模式，则回填对应 UI，否则回填到“自定义”。
- `Switch` 控制启用状态。
- 聊天区新增自动化预览卡片样式，至少展示执行时间摘要、自动化名称、提示词摘要，并提供“创建”入口；整体视觉应与现有工具消息风格保持兼容，但需要明显表达“尚未创建”的待确认状态。

空态和列表态共用同一创建入口，避免分叉出两套交互。

## 风险 / 权衡

- [预设模式与 cron 表达式互转不一致] → 约定持久化以 `cronExpression` 为唯一调度真值，预设模式只作为 UI 辅助，并在编辑回填时使用确定性映射规则。
- [cron 表达式填写错误] → 保存前使用 `node-cron` 校验并在表单中给出明确格式提示，降低错误配置导致的不触发风险。
- [复用固定会话导致上下文不断膨胀] → 将其显式建模为用户主动选择的模式，并默认选择新建会话。
- [应用重启时重复补偿执行] → 用 `lastScheduledAt` 与执行结果回写形成幂等边界，确保同一个计划点不会被重复补偿。
- [自动化执行失败后缺少可观察性] → 至少在列表页展示最近一次失败状态和错误摘要。
- [调度逻辑与 UI/CRUD 混在一起] → 通过 `AutomationService`、`AutomationSchedulerService`、`AutomationRunner` 明确拆分职责。
- [AI 草稿字段不完整导致表单无法打开] → 约定草稿结果的最小必填字段，并在 renderer 缺少关键字段时回退为普通工具结果展示。
- [AI 误改/误删自动化] → 更新与删除工具继续接入现有审批流程，确保写操作需要用户明确批准。

2. 新增主进程 DAO、Service、Controller、shared API 与 preload 暴露。

4. 基于 `node-cron` 实现定时触发注册、cron 表达式校验、运行器和最近一次遗漏补偿。
