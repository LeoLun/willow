## 背景

当前 `packages/core/src/tools/` 中的工具以 `createXTool` 形式分别返回 `AgentTool` 对象，字段结构虽然相近，但没有统一工厂来约束展示信息、权限声明和执行包装。`packages/core/src/tools/index.ts` 直接组装工具列表，`packages/core/src/core-agent.ts` 再把这些工具交给底层 agent runtime。

现有会话流只跟踪 `tool_execution_start`、`tool_execution_update`、`tool_execution_end` 等运行事件，`app/work/src/main/service/session.service.ts` 与前端 `useAgentMessages.ts` 仅把工具调用视为“执行中 / 已完成”，没有“等待用户批准”的中间态。UI 层通过 `packages/ui` 渲染工具调用摘要，但没有决策入口，也没有权限相关元数据。

这次变更同时涉及 tool 定义、主进程执行编排、会话事件模型和前端聊天组件，属于跨模块且带安全语义的改造，适合先明确统一设计。

## 目标 / 非目标

**目标:**
- 引入统一 `createTool` 工厂，收敛 tool 定义字段与扩展点。
- 为工具执行增加统一权限判定与审批暂停机制。
- 让低风险工具默认直通，而不是被统一权限层过度拦截。
- 将 `bash` 的权限控制收敛为“仅高危命令需要人工确认”。
- 首版审批流只实现单次批准/拒绝，不实现会话级记忆授权。
- 在主进程和前端之间补齐待审批工具调用的状态同步。
- 在聊天界面中提供可操作的审批 UI，让用户可以直接批准或拒绝工具调用。

**非目标:**
- 不在本次变更中重做所有 tool renderer 的视觉风格。
- 不引入细粒度的操作系统级沙箱或外部权限服务。
- 不改变模型侧工具调用协议本身，只在现有协议外围增加应用层包装。

## 设计决策

### 决策：在 `packages/core` 中引入统一的工具定义层

新增共享的工具定义工厂与类型，用于生成既兼容 `AgentTool` 运行时、又包含扩展元数据的工具对象。每个工具通过同一个入口声明 `name`、`label`、`description`、`parameters`、`execute` 以及 `permission` 配置，`createAllTools` 只负责装配，不再承担差异化逻辑。

其中 `permission` 不应只支持固定的“总是允许 / 总是审批”，还需要支持由工具基于参数内容做判断。这样 `ls`、`read`、`webfetch`、`websearch` 等低风险工具可以直接放行，而 `bash` 可以在共享契约内挂接自己的高危命令识别逻辑。

备选方案：
- 保持各工具各自返回对象，在注册阶段补一层 metadata map。优点是改动小；缺点是定义和权限策略仍然分散，后续容易漂移。
- 直接在每个 tool 文件内手写统一字段但不抽工厂。优点是简单；缺点是无法强制一致性，也不利于包装公共执行逻辑。

选择工厂方案，因为它既能标准化字段，又能集中接入权限包装。

### 决策：通过统一的审批协调器包装受保护工具的执行流程

在工具执行链路中增加统一的 approval coordinator。工具根据 permission policy 分为直接执行和需审批两类；需审批的工具在真正调用 `execute` 之前先创建待审批状态，由主进程维护一个按 `toolCallId` 索引的 pending approval 容器，并通过会话事件把审批请求同步到 renderer。用户决策返回后，再由 coordinator 恢复执行或构造拒绝结果。

权限判定策略采用“默认允许低风险工具，按需拦截高风险调用”的思路。首版规则中，`ls`、`read`、`webfetch`、`websearch` 默认视为低风险直通；`bash` 不做全量审批，而是先对命令内容做高危匹配，只有命中高危规则时才进入审批流。

首版审批决策只绑定当前 `toolCallId`。也就是说，批准或拒绝只影响当前这一次调用，不提供“始终允许”“本次会话允许”之类的决策范围。

备选方案：
- 在每个敏感 tool 内自行弹出确认。优点是实现局部；缺点是行为不一致，前端无法统一展示，也难以维护跨 tool 的决策状态。
- 在 renderer 里仅做视觉确认，真正执行不阻塞。优点是开发快；缺点是不能提供可靠权限边界。

选择统一 coordinator，因为权限控制必须在执行前集中拦截，才能形成稳定 contract。

### 决策：将审批状态建模为一等会话状态

扩展主进程活动会话流和前端消息状态，使其能表达“待审批”、“已批准”、“已拒绝”等工具调用状态，而不是只依赖正在执行的 `pendingToolCallIds` 集合。审批请求应包含渲染所需的最小上下文，例如 `toolCallId`、tool 名称、参数摘要、权限模式和决策结果。

备选方案：
- 继续只传递 tool execution 事件，前端通过约定推断审批状态。优点是协议变动少；缺点是状态语义脆弱，工具组件难以稳定实现。

选择显式状态建模，因为前后端都需要对审批过程有一致理解。

### 决策：复用现有聊天中的 tool 组件承载审批 UI

前端审批交互优先落在现有聊天消息中的 tool 组件或 renderer 上，保持工具调用与审批结果都出现在同一对话上下文中。待审批工具调用展示明确的等待状态、参数摘要和批准/拒绝按钮；决策后同一组件更新为最终状态，避免跳转到独立设置页或弹窗中心。

由于首版只做单次批准/拒绝，UI 不提供任何持久授权或范围选择控件，只呈现本次调用的批准与拒绝操作。

备选方案：
- 独立做一个全局审批面板。优点是集中；缺点是割裂聊天上下文，用户不容易知道是哪一次调用在等待。
- 使用系统弹窗。优点是显眼；缺点是难以和会话历史对齐，也不适合多工具并发请求。

选择复用聊天内工具组件，因为它最贴近用户当前决策语境，且改造成本较可控。

### 决策：将 `bash` 风险识别放在工具侧实现，统一审批层只负责拦截结果

`bash` 的危险程度依赖具体命令内容，因此高危操作识别应由 `bash` 工具自身或其紧邻辅助模块完成，例如识别删除、大范围覆盖、权限提升、系统级破坏等命令模式。统一审批层只消费“本次调用是否需要审批”的结果并处理后续状态流转，不负责理解全部 shell 语义。

备选方案：
- 在统一权限层硬编码所有 shell 风险规则。优点是集中；缺点是上下文不足，规则会越来越难维护。
- 对所有 `bash` 命令都强制审批。优点是实现简单；缺点是噪声过大，正常低风险命令体验很差。

选择工具侧识别，是因为 `bash` 最了解自己的参数结构，且这能避免把统一权限层变成脚本分析器。

## 风险 / 权衡

- [权限包装侵入现有工具执行路径] → 通过共享工厂和协调器把改动聚焦在注册与执行边界，尽量不修改各工具内部业务逻辑。
- [前后端状态字段扩展后兼容性出错] → 以新增字段为主，保留现有执行事件语义，并在 UI 中对缺省状态做向后兼容处理。
- [审批流阻塞导致工具调用悬挂] → 在主进程维护明确的 pending approval 生命周期，并确保单次批准/拒绝都能结束等待态。
- [参数直接展示可能泄露噪声或过多细节] → UI 只展示适合用户决策的摘要，避免把原始长文本完整铺开。
- [`bash` 高危规则误判过多或过少] → 首版先覆盖明显高危命令模式，并将规则收敛在工具侧，便于后续单独迭代。

## 迁移计划

1. 先引入共享 tool contract，并将现有核心工具迁移到 `createTool` 工厂。
2. 在工具执行链路接入 approval coordinator 和 session 事件扩展。
3. 为主进程与 renderer 补充审批状态传输与响应接口。
4. 改造聊天中的工具组件，展示待审批与已决策状态。
5. 为 `bash` 增加高危命令识别，并确保只有高危命令进入审批流。
6. 验证低风险工具仍可正常执行，`bash` 仅在高危命令下暂停并等待用户决策。

回滚方式：如发现审批链路阻断正常使用，可先移除 coordinator 接入，保留工具工厂重构；UI 增量字段保持可选，不影响旧逻辑渲染。

## 待确认问题

- `bash` 高危命令规则的首版覆盖范围需要在实现时收敛一组最小可用模式，避免一开始规则过宽或过窄。
